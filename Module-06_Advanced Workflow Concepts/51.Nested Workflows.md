## 56. Nested Workflows

**Nested Workflows** means one workflow calls another reusable workflow, and that reusable workflow can call another reusable workflow.

In simple words:

```text
Main Workflow
   ↓
Reusable Workflow 1
   ↓
Reusable Workflow 2
```

Nested workflows are useful when we want to divide a big CI/CD pipeline into small reusable workflows.

## Why use Nested Workflows?

Nested workflows help to:

✅ Reuse common pipelines
✅ Reduce duplicate YAML code
✅ Manage large CI/CD pipelines easily
✅ Keep workflows clean and organized
✅ Standardize build, test, scan, and deploy process

## Important Point

Nested workflows use:

```yaml
on:
  workflow_call:
```

A reusable workflow is called at the **job level**, not inside steps.

Correct:

```yaml
jobs:
  build:
    uses: ./.github/workflows/build.yml
```

Wrong:

```yaml
steps:
  - uses: ./.github/workflows/build.yml
```

## Example Scenario

We will create:

```text
main.yml → calls ci.yml
ci.yml → calls test.yml
```

## 1. Test Reusable Workflow

File:

```text
.github/workflows/test.yml
```

```yaml
name: Test Workflow

on:
  workflow_call:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Run Tests
        run: echo "Running application tests..."
```

## 2. CI Reusable Workflow

File:

```text
.github/workflows/ci.yml
```

```yaml
name: CI Workflow

on:
  workflow_call:

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Build Application
        run: echo "Building application..."

  call-test:
    needs: build
    uses: ./.github/workflows/test.yml
```

## 3. Main Workflow

File:

```text
.github/workflows/main.yml
```

```yaml
name: Main Workflow

on:
  push:
    branches: [ main ]

jobs:
  call-ci:
    uses: ./.github/workflows/ci.yml
```

## How it works

When code is pushed to `main`:

1. `main.yml` starts.
2. It calls `ci.yml`.
3. `ci.yml` runs build job.
4. After build completes, `ci.yml` calls `test.yml`.
5. `test.yml` runs test job.

## Real-Time Java Example

### `java-test.yml`

```yaml
name: Java Test

on:
  workflow_call:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Run Maven Tests
        run: mvn test
```

### `java-ci.yml`

```yaml
name: Java CI

on:
  workflow_call:
    inputs:
      java-version:
        required: true
        type: string

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Java
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: ${{ inputs.java-version }}

      - name: Build Application
        run: mvn clean package

  call-test:
    needs: build
    uses: ./.github/workflows/java-test.yml
```

### `main.yml`

```yaml
name: Main Java Pipeline

on:
  push:
    branches: [ main ]

jobs:
  call-java-ci:
    uses: ./.github/workflows/java-ci.yml
    with:
      java-version: '17'
```

## Explanation Sentence

In this example, the main workflow calls the Java CI workflow, and the Java CI workflow calls the Java Test workflow. This is called nested workflow because workflows are connected in multiple levels.

## Benefits

Reusable pipeline structure.
Clean workflow files.
Easy maintenance.
Good for enterprise CI/CD.
Useful for build, test, scan, and deploy separation.

## Best Practices

Use clear workflow names.
Keep each workflow focused on one purpose.
Pass inputs properly between workflows.
Use secrets carefully.
Avoid too many nesting levels.
Use version tags when calling workflows from another repository.

## Simple Definition

**Nested Workflows in GitHub Actions means calling reusable workflows inside other reusable workflows to create a multi-level CI/CD pipeline.**
