## 52. Dynamic Matrix Workflows

**Dynamic Matrix Workflow** means creating matrix values dynamically during workflow execution instead of hardcoding them directly in the YAML file.

Normal matrix:

```yaml
matrix:
  java-version: [17, 21]
```

Dynamic matrix:

```yaml
matrix:
  java-version: ${{ fromJson(needs.prepare.outputs.versions) }}
```

## Why use Dynamic Matrix?

Dynamic matrix is useful when matrix values come from:

Project files
API response
Script output
Changed folders
Environment list
Configuration file

## Example Scenario

Suppose we want to test only selected Java versions, and those versions are generated by a first job.

## Full Example

```yaml
name: Dynamic Matrix Example

on:
  push:
    branches: [ main ]

jobs:
  prepare:
    runs-on: ubuntu-latest

    outputs:
      versions: ${{ steps.set-matrix.outputs.versions }}

    steps:
      - name: Generate Java Versions
        id: set-matrix
        run: |
          echo 'versions=["17","21"]' >> $GITHUB_OUTPUT

  test:
    needs: prepare
    runs-on: ubuntu-latest

    strategy:
      matrix:
        java-version: ${{ fromJson(needs.prepare.outputs.versions) }}

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Java
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: ${{ matrix.java-version }}

      - name: Run Tests
        run: mvn test
```

## Explanation

First job:

```yaml
prepare:
```

generates this output:

```json
["17","21"]
```

Second job:

```yaml
test:
```

uses that output as matrix values.

So the workflow runs:

```text
Test with Java 17
Test with Java 21
```

## Important Line

```yaml
fromJson(needs.prepare.outputs.versions)
```

This converts JSON text into matrix values.

Without `fromJson`, GitHub treats it like normal text.

## Real-Time Example: Test Changed Microservices

Suppose repository has:

```text
services/user-service
services/order-service
services/payment-service
```

We want to test only changed services.

```yaml
name: Dynamic Service Matrix

on: push

jobs:
  detect-services:
    runs-on: ubuntu-latest

    outputs:
      services: ${{ steps.set.outputs.services }}

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Detect Changed Services
        id: set
        run: |
          echo 'services=["user-service","payment-service"]' >> $GITHUB_OUTPUT

  test-services:
    needs: detect-services
    runs-on: ubuntu-latest

    strategy:
      matrix:
        service: ${{ fromJson(needs.detect-services.outputs.services) }}

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Test Service
        run: |
          echo "Testing ${{ matrix.service }}"
          cd services/${{ matrix.service }}
          mvn test
```

## Explanation Sentence

In this workflow, the first job detects which services are changed, and the second job dynamically creates matrix jobs only for those changed services.

## Benefits

Saves time.
Avoids hardcoded values.
Useful for monorepos.
Runs only required jobs.
Makes workflow more flexible.

## Simple Definition

**Dynamic Matrix Workflow means generating matrix values during workflow runtime and using them to run jobs dynamically.**
