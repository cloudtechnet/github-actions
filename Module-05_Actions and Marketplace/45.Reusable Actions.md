## 45. Reusable Actions

**Reusable Actions** means creating one action and using it again in multiple workflows or multiple projects.

It helps avoid writing the same code again and again.

## Why use Reusable Actions?

Reusable actions help to:

✅ Save time
✅ Reduce duplicate code
✅ Keep workflows clean
✅ Follow same process in all projects
✅ Make CI/CD easier to maintain

## Types of Reusable Actions

Reusable actions can be:

1. **Composite Actions**
2. **JavaScript Actions**
3. **Docker Actions**
4. **Marketplace Actions**

## Example: Reusable Composite Action

### Folder Structure

```text
.github/actions/java-build/
└── action.yml
```

## Step 1: Create reusable action

```yaml
name: Java Build Reusable Action
description: Setup Java and build Maven project

inputs:
  java-version:
    description: Java version
    required: true
    default: '17'

runs:
  using: composite
  steps:
    - name: Setup Java
      uses: actions/setup-java@v4
      with:
        distribution: temurin
        java-version: ${{ inputs.java-version }}

    - name: Maven Build
      shell: bash
      run: mvn clean install
```

## Step 2: Use reusable action in workflow

```yaml
name: Java CI

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Run Reusable Java Build Action
        uses: ./.github/actions/java-build
        with:
          java-version: '17'
```

## Explanation

When code is pushed:

1. Workflow starts.
2. Repository code is checked out.
3. Reusable action is called.
4. Java 17 is installed.
5. Maven build runs.
6. Same action can be reused in other workflows.

## Reusing from another repository

If action is stored in another repository:

```yaml
- name: Use Company Java Build Action
  uses: company-name/github-actions/java-build@v1
  with:
    java-version: '17'
```

## Real-Time Use Case

A company has 10 Java projects.

Instead of writing Java setup and Maven build steps in every project, they create one reusable action:

```text
company-actions/java-build
```

Then all projects use:

```yaml
uses: company-actions/java-build@v1
```

So if build logic changes, update only one action.

## Reusable Action vs Reusable Workflow

| Reusable Action               | Reusable Workflow             |
| ----------------------------- | ----------------------------- |
| Reuses steps                  | Reuses full workflow/job      |
| Called inside `steps`         | Called inside `jobs`          |
| Uses `uses:` in step          | Uses `uses:` at job level     |
| Good for small repeated tasks | Good for full CI/CD pipelines |

## Best Practices

Use clear action names.
Add inputs for flexibility.
Use version tags like `@v1`.
Keep actions small and focused.
Do not hardcode secrets.
Write README documentation.
Test before using in production.

## Simple Definition

**Reusable Actions are custom or marketplace actions that can be used multiple times across workflows and projects to reduce repeated code and standardize automation.**
