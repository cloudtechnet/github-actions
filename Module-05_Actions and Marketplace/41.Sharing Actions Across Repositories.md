## 46. Sharing Actions Across Repositories

**Sharing Actions Across Repositories** means creating one custom GitHub Action in one repository and using it in many other repositories.

This is useful for companies, teams, and training projects where the same build, test, scan, or deployment logic is needed again and again.

## Why share actions across repositories?

✅ Avoid repeated workflow code
✅ Maintain one common action
✅ Standard process for all projects
✅ Easy updates
✅ Better team collaboration

## Example Scenario

Suppose we have many Java projects:

```text
student-app
employee-app
payment-service
order-service
```

All projects need the same steps:

```text
Checkout code
Setup Java
Run Maven build
```

Instead of writing the same steps in every repository, we create one shared action.

## Repository 1: Shared Action Repository

Create one repository:

```text
cloudtechnet/shared-github-actions
```

Folder structure:

```text
shared-github-actions/
└── java-build/
    └── action.yml
```

### `java-build/action.yml`

```yaml
name: Shared Java Build Action
description: Setup Java and build Maven project

inputs:
  java-version:
    description: Java version to use
    required: true
    default: '17'

runs:
  using: composite
  steps:
    - name: Setup Java
      uses: actions/setup-java@v4
      with:
        distribution: temurin
        java-version: ${{ inputs.java-version }}

    - name: Run Maven Build
      shell: bash
      run: mvn clean install
```

## Repository 2: Project Repository

In another repository, create workflow:

```text
.github/workflows/build.yml
```

```yaml
name: Java CI Using Shared Action

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Project Code
        uses: actions/checkout@v4

      - name: Use Shared Java Build Action
        uses: cloudtechnet/shared-github-actions/java-build@v1
        with:
          java-version: '17'
```

## Explanation

Here:

```yaml
uses: cloudtechnet/shared-github-actions/java-build@v1
```

Means:

```text
cloudtechnet        → GitHub organization/user
shared-github-actions → repository name
java-build          → folder where action.yml is present
v1                  → version tag
```

## How to share properly

### Step 1: Create shared action repo

Example:

```text
shared-github-actions
```

### Step 2: Add custom action files

Example:

```text
java-build/action.yml
```

### Step 3: Commit and push code

### Step 4: Create version tag

```bash
git tag v1
git push origin v1
```

### Step 5: Use in other repos

```yaml
uses: username/shared-github-actions/java-build@v1
```

## Example: Shared Docker Build Action

Folder:

```text
shared-github-actions/
└── docker-build/
    └── action.yml
```

```yaml
name: Shared Docker Build Action
description: Build Docker image

inputs:
  image-name:
    description: Docker image name
    required: true

runs:
  using: composite
  steps:
    - name: Build Docker Image
      shell: bash
      run: docker build -t ${{ inputs.image-name }} .
```

Use it in another repo:

```yaml
name: Docker CI

on: push

jobs:
  docker-build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Use Shared Docker Build Action
        uses: cloudtechnet/shared-github-actions/docker-build@v1
        with:
          image-name: myapp:latest
```

## Public vs Private Sharing

### Public repository

Any GitHub repository can use the action:

```yaml
uses: username/repo-name/action-folder@v1
```

### Private repository

Only repositories with permission can use it.
For organizations, access must be allowed from repository or organization settings.

## Best Practices

Use version tags like `@v1`, `@v2`.
Do not use `@main` in production.
Write clear README documentation.
Keep actions small and reusable.
Use inputs for flexible values.
Do not hardcode secrets.
Store secrets in the project repository using GitHub Secrets.

## Simple Definition

**Sharing Actions Across Repositories means creating one reusable GitHub Action in one repository and calling it from other repositories using `uses:`.**
