## 47. Versioning Actions

**Versioning Actions** means using a specific version of an action in your workflow.

Example:

```yaml
uses: actions/checkout@v4
```

Here, `@v4` is the action version.

## Why Versioning is Important

Versioning helps to:

✅ Keep workflow stable
✅ Avoid sudden breaking changes
✅ Use tested action versions
✅ Track updates easily
✅ Improve security and reliability

## Version Types

### 1. Branch Version

```yaml
uses: actions/checkout@main
```

This uses the latest code from `main`.

Not recommended for production because changes may break your workflow.

### 2. Tag Version

```yaml
uses: actions/checkout@v4
```

This uses a released version.

Recommended for normal projects.

### 3. Commit SHA Version

```yaml
uses: actions/checkout@8ade135a41bc03ea155e62e844d188df1ea18608
```

This locks the action to one exact commit.

Most secure option.

## Example: Java Build Workflow

```yaml
name: Java Build with Versioned Actions

on:
  push:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup Java
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 17

      - name: Build Project
        run: mvn clean install
```

## Explanation

```yaml
uses: actions/checkout@v4
```

Uses version 4 of checkout action.

```yaml
uses: actions/setup-java@v4
```

Uses version 4 of Java setup action.

This makes the workflow more stable than using `@main`.

## Custom Action Versioning Example

Create tag for your custom action:

```bash
git tag v1
git push origin v1
```

Use it in another repository:

```yaml
uses: cloudtechnet/shared-github-actions/java-build@v1
```

## Updating Versions

Old:

```yaml
uses: actions/checkout@v3
```

New:

```yaml
uses: actions/checkout@v4
```

Before updating, check action documentation and test the workflow.

## Best Practices

Use version tags like `@v1`, `@v4`.
Avoid using `@main` in production.
Use commit SHA for high-security projects.
Test before upgrading action versions.
Update old versions regularly.
Read release notes before changing versions.

## Simple Definition

**Versioning Actions means selecting a fixed version of a GitHub Action using tags, branches, or commit SHA to keep workflows stable and secure.**
