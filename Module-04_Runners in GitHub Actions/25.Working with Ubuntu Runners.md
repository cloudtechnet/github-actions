# 25. Working with Ubuntu Runners

Ubuntu runners are the most commonly used runners in GitHub Actions.

They are:

* Fast
* Stable
* Pre-configured
* Developer-friendly
* Best for CI/CD pipelines

Most GitHub Actions workflows use:

```yaml id="25e7hz"
runs-on: ubuntu-latest
```

---

# What is an Ubuntu Runner?

An Ubuntu runner is:

> A Linux-based virtual machine provided by GitHub to execute workflow jobs.

GitHub creates a temporary Ubuntu VM whenever a workflow starts.

---

# Common Ubuntu Runner Labels

| Label         | Description          |
| ------------- | -------------------- |
| ubuntu-latest | Latest stable Ubuntu |
| ubuntu-24.04  | Ubuntu 24.04         |
| ubuntu-22.04  | Ubuntu 22.04         |
| ubuntu-20.04  | Ubuntu 20.04         |

---

# Example

```yaml id="jlwmv6"
runs-on: ubuntu-latest
```

---

# Why Ubuntu Runners are Popular

Ubuntu runners provide:

| Advantage              | Description              |
| ---------------------- | ------------------------ |
| Fast startup           | Quick workflow execution |
| Linux ecosystem        | Best for DevOps tools    |
| Docker support         | Container workflows      |
| Pre-installed software | Saves setup time         |
| Good compatibility     | Works with most tools    |

---

# Ubuntu Runner Workflow Lifecycle

```text id="4m0d0q"
Workflow Triggered
        ↓
Ubuntu VM Created
        ↓
Repository Downloaded
        ↓
Workflow Steps Executed
        ↓
Logs Generated
        ↓
VM Destroyed
```

---

# Important Characteristics

| Feature             | Description                 |
| ------------------- | --------------------------- |
| Temporary VM        | Fresh environment every run |
| Isolated execution  | Secure workflow execution   |
| Cloud-hosted        | Managed by GitHub           |
| Linux shell support | Bash commands available     |

# 1. Basic Ubuntu Runner Example

---

# Workflow

```yaml id="7fey5t"
name: Ubuntu Runner Demo

on: push

jobs:
  demo:
    runs-on: ubuntu-latest

    steps:
      - name: Display Message
        run: echo "Hello from Ubuntu Runner"
```

| Step | Action                   |
| ---- | ------------------------ |
| 1    | GitHub creates Ubuntu VM |
| 2    | Workflow executes        |
| 3    | Message displayed        |
| 4    | VM removed               |

# 2. Checkout Repository on Ubuntu Runner

Most workflows first download repository code.

Use:

```yaml id="jlwm4h"
actions/checkout
```

---

# Example

```yaml id="t2m44m"
name: Checkout Example

on: push

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: List Files
        run: ls -la
```

---

# Explanation

| Step     | Purpose              |
| -------- | -------------------- |
| checkout | Downloads repository |
| ls -la   | Displays files       |

---

# Sample Output


# 3. Running Linux Commands

Ubuntu runners support Linux shell commands.

---

# Example Commands

| Command | Purpose           |
| ------- | ----------------- |
| `pwd`   | Current directory |
| `ls`    | List files        |
| `mkdir` | Create directory  |
| `cat`   | Read file         |
| `echo`  | Print text        |

---

# Example Workflow

```yaml id="1kztz7"
jobs:
  linux-demo:
    runs-on: ubuntu-latest

    steps:
      - run: pwd

      - run: ls

      - run: mkdir test-folder

      - run: echo "Ubuntu Runner" > file.txt

      - run: cat file.txt
```

---

# Output

```text id="3krw0t"

Ubuntu Runner
```

# 5. Installing Packages in Ubuntu Runner

Ubuntu runners allow package installation using:

```bash

```

---

# Example

```yaml id="jlwm9k"
  - name: Install Tree Package

      sudo apt-get update
```

# 11. Parallel Jobs with Ubuntu Runners

Each job gets separate Ubuntu VM.

---

# Example

```yaml id="cshyey"

  frontend:
    runs-on: ubuntu-latest

  backend:
    runs-on: ubuntu-latest
```

---

# Execution

```text id="k8i8y4"

backend  → Ubuntu Runner 2
```

Both run simultaneously.

# 12. Matrix Builds with Ubuntu Runners

---

# Example

```yaml id="y8jlwm"
jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node: [16, 18, 20]

    steps:
      - run: echo "Testing Node ${{ matrix.node }}"
```

---

# Result

GitHub creates:

| Runner   | Node Version |
| -------- | ------------ |
| Runner 1 | 16           |
| Runner 2 | 18           |
| Runner 3 | 20           |

# 14. Real-World CI/CD Example

---

# Scenario

Build and deploy Node.js application.

---

# Workflow

```yaml id="jlwm77"
name: Ubuntu CI/CD

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Install Dependencies
        run: npm install

      - name: Run Tests
        run: npm test

      - name: Build Project
        run: npm run build

  deploy:
    needs: build

    runs-on: ubuntu-latest

    steps:
      - name: Deploy Application
        run: echo "Deploying application"
```

---

# Execution Flow

```text id="jlwm7s"
Push Code
    ↓
Ubuntu Runner Created
    ↓
Build + Test
    ↓
Deploy Job
    ↓
Runner Destroyed
```

# 16. Limitations of Ubuntu Runners

| Limitation            | Description               |
| --------------------- | ------------------------- |
| Temporary environment | Data removed after run    |
| Runtime limits        | Max execution time        |
| Limited customization | Cannot permanently modify |
| Shared infrastructure | Limited resources         |

# 17. Best Practices

| Best Practice               | Reason                    |
| --------------------------- | ------------------------- |
| Use ubuntu-latest           | Best compatibility        |
| Cache dependencies          | Faster workflows          |
| Keep jobs optimized         | Reduce execution time     |
| Use matrix builds carefully | Avoid unnecessary runners |
| Store artifacts             | Preserve files            |

---

# 18. Common Mistakes

---

# Mistake 1 – Assuming Persistent Storage

Files disappear after workflow ends.

---

# Mistake 2 – Hardcoding Paths

Avoid fixed Linux paths when possible.

---

# Mistake 3 – Installing Everything Repeatedly

Can slow pipelines.

Use caching.

---

# Mistake 4 – Ignoring Runner Limits

Large builds may hit timeout/resource limits.
