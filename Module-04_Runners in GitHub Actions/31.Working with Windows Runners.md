# 31. Working with Windows Runners

Windows runners in GitHub Actions are virtual machines running Microsoft Windows.

They are mainly used for:

* .NET applications
* Windows software builds
* PowerShell scripts
* Desktop applications
* Windows-specific testing

GitHub provides fully managed Windows virtual machines for workflow execution.

---

# What is a Windows Runner?

A Windows runner is:

> A Windows-based virtual machine used to execute GitHub Actions workflows.

GitHub automatically:

* Creates Windows VM
* Runs workflow jobs
* Removes VM after completion

---

# Common Windows Runner Labels

| Label          | Description                  |
| -------------- | ---------------------------- |
| windows-latest | Latest stable Windows runner |
| windows-2022   | Windows Server 2022          |
| windows-2019   | Windows Server 2019          |

---

# 14. Real-World Windows CI/CD Example

---

# Scenario

Build and test ASP.NET application.

---

# Workflow

```yaml id="jlwmif"
name: ASP.NET CI/CD

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: windows-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup .NET
        uses: actions/setup-dotnet@v4

        with:
          dotnet-version: 8.0

      - name: Restore Packages
        run: dotnet restore

      - name: Build Project
        run: dotnet build --configuration Release

      - name: Run Tests
        run: dotnet test
```

# 16. Limitations of Windows Runners

| Limitation                  | Description           |
| --------------------------- | --------------------- |
| Slower startup              | Compared to Ubuntu    |
| Higher resource usage       | Larger VM size        |
| Longer build times          | Windows overhead      |
| Limited Linux compatibility | Some Linux tools fail |

---

# 17. Windows Runner vs Ubuntu Runner

| Feature        | Windows            | Ubuntu         |
| -------------- | ------------------ | -------------- |
| Speed          | Slower             | Faster         |
| Default Shell  | PowerShell         | Bash           |
| Best For       | .NET, Windows apps | DevOps, Docker |
| Resource Usage | Higher             | Lower          |
| Popularity     | Medium             | Very High      |

---

# 18. Best Practices

| Best Practice                 | Reason                  |
| ----------------------------- | ----------------------- |
| Use Windows only when needed  | Ubuntu is faster        |
| Use PowerShell properly       | Native Windows support  |
| Cache dependencies            | Improve performance     |
| Test cross-platform carefully | OS differences matter   |
| Use matrix builds wisely      | Avoid excessive runners |

# 19. Common Mistakes

---

# Mistake 1 – Using Linux Commands

Wrong on Windows:

```yaml id="jlwmx1"
run: ls
```

Use:

```yaml id="jlwmu3"
run: dir
```

---

# Mistake 2 – Wrong Environment Variable Syntax

Wrong:

```yaml id="jlwm0g"
echo $APP_NAME
```

Correct:

```yaml id="jlwmy1"
echo $env:APP_NAME
```

---

# Mistake 3 – Assuming Bash Default

Windows runners default to PowerShell.

---

# Mistake 4 – Ignoring Path Differences

Windows paths use:

```text id="jlwm1f"
\
```

instead of:

```text id="jlwmxk"
/
```

# 20. Advanced Example – Multi Platform Testing

```yaml id="jlwm2l"
name: Cross Platform Testing

on: push

jobs:
  test:
    strategy:
      matrix:
        os:
          - ubuntu-latest
          - windows-latest

    runs-on: ${{ matrix.os }}

      - uses: actions/checkout@v4

      - name: Display OS
        run: echo "Running on $RUNNER_OS"
```
