# GitHub Actions – Run Workflows on Self-Hosted Runners

Once a self-hosted runner is installed and connected (Linux/Windows/macOS), the next step is to **run workflows on it**.

In GitHub Actions, this is controlled using:

```yaml
runs-on: self-hosted
```

This tells GitHub:

> “Do NOT use GitHub-hosted machines. Use my own machine (self-hosted runner).”

---

# How Workflow Execution Works

```text id="jlwmx1"
GitHub Repo (Workflow Trigger)
            ↓
GitHub Actions Queue
            ↓
Find Matching Self-Hosted Runner
            ↓
Runner Accepts Job
            ↓
Steps Execute on Your Machine
            ↓
Logs Sent Back to GitHub
```

---

# Key Concept: Runner Matching

GitHub sends jobs based on **labels**.

| Label Type   | Example                 |
| ------------ | ----------------------- |
| Required     | self-hosted             |
| OS label     | linux / windows / macos |
| Custom label | production / gpu / test |

---

# 1. Basic Workflow on Self-Hosted Runner

---

## Example

```yaml id="jlwmp1"
name: Basic Self Hosted Workflow

on: push

jobs:
  demo:
    runs-on: self-hosted

    steps:
      - name: Print Message
        run: echo "Hello from Self Hosted Runner"
```

---

## What Happens?

| Step | Action                       |
| ---- | ---------------------------- |
| 1    | GitHub triggers workflow     |
| 2    | Finds any self-hosted runner |
| 3    | Sends job to runner          |
| 4    | Executes `echo`              |
| 5    | Sends logs back              |

---

## Output

```text id="jlwmq2"
Hello from Self Hosted Runner
```

# 2. Running on Specific OS Self-Hosted Runners

If you have multiple runners:

* Linux runner
* Windows runner
* macOS runner

You can control execution.

---

## Example (Linux Only)

```yaml id="jlwmr3"
jobs:
  build:
    runs-on:
      - self-hosted
      - linux

    steps:
      - name: Linux Command
        run: uname -a
```

---

## Example (Windows Only)

```yaml id="jlwms4"
jobs:
  build:
    runs-on:
      - self-hosted
      - windows

    steps:
      - name: Windows Command
        run: echo %OS%
```

---

## Example (macOS Only)

```yaml id="jlwmt5"
jobs:
  build:
    runs-on:
      - self-hosted
      - macos

    steps:
      - name: macOS Command
        run: uname -a
```

# 3. Using Custom Labels (Very Important)

You can assign labels when configuring runner:

Examples:

* production
* staging
* gpu
* high-memory

---

## Example Workflow

```yaml id="jlwmu6"
jobs:
  deploy:
    runs-on:
      - self-hosted
      - linux
      - production

    steps:
      - name: Deploy App
        run: echo "Deploying to Production Server"
```

---

## Meaning

This job will ONLY run on a runner that has:

* self-hosted
* linux
* production

# 4. Real-World Example: CI Pipeline

---

## Scenario

Node.js application build + test on self-hosted runner.

---

## Workflow

```yaml id="jlwmv7"
name: Node CI on Self Hosted Runner

on: push

jobs:
  build:
    runs-on:
      - self-hosted
      - linux

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Install Dependencies
        run: npm install

      - name: Run Tests
        run: npm test
```

---

## Execution Flow

```text id="jlwmw8"
Push Code
   ↓
GitHub Workflow Triggered
   ↓
Self Hosted Linux Runner Selected
   ↓
npm install
   ↓
npm test
   ↓
Results Sent to GitHub
```
