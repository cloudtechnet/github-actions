# 32. Runner Security Best Practices

Self-hosted runners give **full control over the machine**, but that also means **full responsibility for security**.

If not configured properly, a compromised workflow can:

* Access your system files
* Steal secrets
* Modify deployments
* Attack internal networks

So security is one of the MOST important topics in self-hosted runners.

---

# 1. Why Runner Security Matters

GitHub-hosted runners are isolated and temporary.

But self-hosted runners:

> Run on YOUR machine, YOUR network, YOUR permissions.

---

## Risk Example

```text id="jlwm1a"
Malicious workflow → Self-hosted runner → Access production server → Data leak
```

---

# 2. Golden Rule of Runner Security

> NEVER trust external code on a powerful self-hosted runner without restrictions.

---

# 3. Best Practice 1 – Use Dedicated Machines

Do NOT use:

* Personal laptop ❌
* Production database server ❌
* Shared company server ❌

Use:

* Dedicated CI server ✔
* Isolated VM ✔
* Container-based runner ✔

---

## Example Setup

```text id="jlwm2b"
Runner Machine:
  - No user login usage
  - Only CI/CD role
  - No sensitive personal files
```

---

# 4. Best Practice 2 – Use Runner Groups (Access Control)

Runner groups restrict which repositories can use a runner.

---

## Example

| Group      | Access               |
| ---------- | -------------------- |
| Dev Group  | All repositories     |
| Prod Group | Only production repo |

---

## Why Important?

Prevents:

```text id="jlwm3c"
Untrusted repo → using production runner ❌
```

---

# 5. Best Practice 3 – Restrict Repository Access

Only allow trusted repositories.

---

## GitHub Setting:

```text id="jlwm4d"
Settings → Actions → Runner Groups → Repository Access
```

---

## Rule:

* Public repos → NEVER use production runners
* Internal repos → controlled access only

---

# 6. Best Practice 4 – Use Ephemeral Runners (Recommended)

Ephemeral runner = destroyed after one job.

---

## Benefit:

```text id="jlwm5e"
Fresh environment every time → No leftover data
```

---

## Example Concept

```text id="jlwm6f"
Job starts → Runner used → Job ends → Runner destroyed
```

---

# 7. Best Practice 5 – Avoid Running Sensitive Workloads on Public Repo

If repo is public:

* Anyone can open PR
* Attacker can inject code

---

## Dangerous Example ❌

```yaml id="jlwm7g"
on: pull_request

jobs:
  build:
    runs-on: self-hosted
```

---

## Why dangerous?

PR from attacker → code executes on your machine.

---

# 8. Best Practice 6 – Use Least Privilege Principle

Only give minimum permissions.

---

## Example

Instead of:

```yaml id="jlwm8h"
permissions: write-all
```

Use:

```yaml id="jlwm9i"
permissions:
  contents: read
```

---

# 9. Best Practice 7 – Secure Secrets Handling

Never expose secrets in logs.

---

## Bad Example ❌

```yaml id="jlwm0j"
- run: echo $SECRET_KEY
```

---

## Good Example ✔

```yaml id="jlwm1k"
- name: Use Secret Safely
  run: |
    curl -H "Authorization: $SECRET_KEY" https://api.example.com
```

---

## GitHub Secret Storage

```text id="jlwm2l"
Settings → Secrets and Variables → Actions
```

---

# 10. Best Practice 8 – Run Runner as a Service (Safe Mode)

Instead of manual execution:

```bash id="jlwm3m"
./run.sh
```

Use service mode:

```bash id="jlwm4n"
./svc.sh install
./svc.sh start
```

---

## Benefit:

* Auto restart after reboot
* Controlled execution
* More stable

---

# 11. Best Practice 9 – Keep Runner Updated

Old runners may have vulnerabilities.

---

## Update Process

```bash id="jlwm5o"
Stop runner
Download latest version
Reconfigure runner
Start again
```

---

# 12. Best Practice 10 – Network Isolation

Self-hosted runner should NOT have full network access.

---

## Recommended Setup:

```text id="jlwm6p"
Runner → Private subnet → Controlled access only
```

---

## Avoid:

* Direct internet + production DB access ❌

---

# 13. Best Practice 11 – Avoid Running on Pull Requests (Public Repos)

Pull requests are HIGH RISK.

---

## Safe Workflow

```yaml id="jlwm7q"
on:
  push:
    branches:
      - main
```

---

## Unsafe Workflow ❌

```yaml id="jlwm8r"
on: pull_request
```

---

# 14. Best Practice 12 – Logging and Monitoring

Always monitor runner activity.

---

## What to monitor:

* CPU usage
* Disk usage
* Unexpected commands
* Failed jobs

---

## Example Tools:

* Prometheus
* Grafana
* Windows Event Logs
* Linux syslog

---

# 15. Best Practice 13 – Use Separate Runners for Different Environments

---

## Example Setup

| Runner      | Purpose            |
| ----------- | ------------------ |
| dev-runner  | Development builds |
| qa-runner   | Testing            |
| prod-runner | Deployment         |

---

## Workflow

```yaml id="jlwm9s"
runs-on:
  - self-hosted
  - production
```

---

# 16. Best Practice 14 – Disable Unused Runners

Unused runners increase attack surface.

---

## Command

```bash id="jlwm0t"
./config.sh remove
```

---

# 17. Best Practice 15 – Avoid Storing Secrets Locally

Never store:

* API keys
* Passwords
* Tokens

inside runner machine.

---

## Use GitHub Secrets instead

---

# 18. Real-World Secure Workflow Example

```yaml id="jlwm1u"
name: Secure CI Pipeline

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on:
      - self-hosted
      - linux

    permissions:
      contents: read

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Build Application
        run: npm install && npm run build

  deploy:
    needs: build
    runs-on:
      - self-hosted
      - linux
      - production

    steps:
      - name: Deploy Safely
        run: echo "Deploying to secure environment"
```

# Final Understanding

Runner security in GitHub Actions is critical because self-hosted runners:

> Run real code on real machines inside your network.

So security best practices ensure:

* Safe CI/CD execution
* Protection of internal systems
* Prevention of malicious workflows
* Controlled access to infrastructure
* Stable enterprise automation

👉 In simple terms:

**Secure runner = secure entire DevOps pipeline**
