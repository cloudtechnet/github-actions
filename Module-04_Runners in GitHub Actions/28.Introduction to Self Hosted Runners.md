# 33. Introduction to Self-Hosted Runners

In GitHub Actions, workflows need a machine to execute jobs.

GitHub provides two types of runners:

| Runner Type          | Managed By        |
| -------------------- | ----------------- |
| GitHub Hosted Runner | GitHub            |
| Self-Hosted Runner   | User/Organization |

A Self-Hosted Runner is:

> A machine that you manage yourself and connect to GitHub Actions for executing workflows.

This machine can be:

* Physical server
* Virtual machine
* Cloud VM
* Local computer
* Docker container

---

# What is a Self-Hosted Runner?

A self-hosted runner is your own system that runs GitHub Actions jobs.

Instead of GitHub creating temporary virtual machines, your machine executes workflow jobs.

---

# Simple Architecture

```text id="jlwm5z"
Developer Pushes Code
          ↓
GitHub Actions Triggered
          ↓
Self-Hosted Runner Receives Job
          ↓
Workflow Executes on Your Machine
          ↓
Results Sent Back to GitHub
```

---

# Why Use Self-Hosted Runners?

Organizations use self-hosted runners when they need:

* More control
* Custom environments
* Internal network access
* Specialized hardware
* Faster execution
* Reduced GitHub runner costs

---

# GitHub Hosted vs Self-Hosted Runners

| Feature                 | GitHub Hosted | Self-Hosted         |
| ----------------------- | ------------- | ------------------- |
| Managed by              | GitHub        | User                |
| Setup                   | Automatic     | Manual              |
| Infrastructure control  | Limited       | Full                |
| Cost                    | Usage-based   | Your infrastructure |
| Persistent environment  | No            | Yes                 |
| Internal network access | Limited       | Full                |
| Custom software         | Limited       | Unlimited           |

# 2. Registering a Self-Hosted Runner

To use a self-hosted runner:

1. Open GitHub repository
2. Go to:

   ```text
   Settings → Actions → Runners
   ```
3. Click:

   ```text
   New self-hosted runner
   ```
4. Choose operating system
5. Download runner package
6. Configure runner
7. Start runner

---

# Linux Runner Setup Example

GitHub provides commands like:

```bash id="jlwm3x"
mkdir actions-runner

cd actions-runner

curl -o actions-runner-linux-x64.tar.gz -L <download-url>

tar xzf ./actions-runner-linux-x64.tar.gz
```

---

# Configure Runner

```bash id="jlwmfu"
./config.sh --url https://github.com/owner/repo --token TOKEN
```

---

# Start Runner

```bash id="jlwm0o"
./run.sh
```

# 10. Security Considerations

Self-hosted runners require careful security management.

---

# Important Risks

| Risk                  | Explanation              |
| --------------------- | ------------------------ |
| Malicious workflows   | Can affect machine       |
| Secret exposure       | Local environment risk   |
| Persistent compromise | Machine remains active   |
| Network access        | Internal systems exposed |

---

# Security Best Practices

| Best Practice              | Reason           |
| -------------------------- | ---------------- |
| Use isolated machines      | Reduce risks     |
| Restrict repository access | Security control |
| Avoid public repositories  | Prevent abuse    |
| Regular updates            | Security patches |
| Use ephemeral runners      | Better isolation |

---

# Important Warning

Never use sensitive production servers directly as runners.

# 12. Real-World CI/CD Example

---

# Scenario

Company deploys application to internal Kubernetes cluster.

GitHub-hosted runners cannot access private network.

Use self-hosted runner.

---

# Workflow

```yaml id="jlwmoj"
name: Internal Deployment

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on:
      - self-hosted
      - linux
      - production

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Build Application
        run: npm install

      - name: Deploy to Kubernetes
        run: kubectl apply -f deployment.yaml
```

---

# Execution Flow

```text id="jlwm2w"
Push Code
    ↓
GitHub Actions Triggered
    ↓
Self-Hosted Runner Receives Job
    ↓
Build Application
    ↓
Deploy to Internal Kubernetes
```
