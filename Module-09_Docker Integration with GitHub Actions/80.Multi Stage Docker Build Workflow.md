# 80. Multi-Stage Docker Build Workflow

### What is a Multi-Stage Docker Build?

A **Multi-Stage Docker Build** is a Docker build technique that uses multiple `FROM` statements in a single Dockerfile. Each stage performs a specific task, such as building the application or creating the final runtime image.

This approach helps create **smaller, faster, and more secure Docker images** by including only the files needed to run the application.

---

## Why Use Multi-Stage Docker Builds?

Without multi-stage builds, the final Docker image contains build tools, source code, and temporary files, making it unnecessarily large.

❌ Single-Stage Build

```text
Dockerfile
        ↓
Install Build Tools
        ↓
Compile Application
        ↓
Include Source Code
        ↓
Large Docker Image
```

The final image contains everything, even files that are not needed at runtime.

---

✅ Multi-Stage Build

```text
Build Stage
        ↓
Compile Application
        ↓
Copy Only Required Files
        ↓
Runtime Stage
        ↓
Small and Optimized Image
```

Only the compiled application and runtime dependencies are included in the final image.

---

## Benefits of Multi-Stage Builds

- Smaller Docker images
- Faster image downloads
- Reduced attack surface
- Better security
- Faster deployments
- Cleaner Dockerfiles

---

## Prerequisites

Before using multi-stage builds, ensure you have:

- Docker installed
- A Dockerfile
- Application source code
- GitHub repository
- GitHub Actions enabled

## Single-Stage Dockerfile Example

```dockerfile
FROM node:20

WORKDIR /app

COPY . .

RUN npm install

RUN npm run build

EXPOSE 3000

CMD ["npm", "start"]
```

### Problem

The final image contains:

- Node.js build tools
- Source code
- Development dependencies
- Build cache

Result:

```text
Large Docker Image
```

---

## Multi-Stage Dockerfile Example

```dockerfile
# Stage 1 - Build Stage
FROM node:20 AS builder

WORKDIR /app

COPY package*.json ./

RUN npm install

COPY . .

RUN npm run build

# Stage 2 - Runtime Stage
FROM node:20-alpine

WORKDIR /app

COPY --from=builder /app .

EXPOSE 3000

CMD ["npm", "start"]
```

---

## Understanding the Dockerfile

### Stage 1

```dockerfile
FROM node:20 AS builder
```

Creates the **builder stage**.

This stage:

- Installs dependencies
- Compiles the application
- Produces build artifacts

---

### Working Directory

```dockerfile
WORKDIR /app
```

Sets the working directory inside the container.

---

### Copy Dependency Files

```dockerfile
COPY package*.json ./
```

Copies the package files.

This helps Docker cache dependency installation when the source code changes.

---

### Install Dependencies

```dockerfile
RUN npm install
```

Installs project dependencies.

---

### Copy Source Code

```dockerfile
COPY . .
```

Copies the application source code.

---

### Build Application

```dockerfile
RUN npm run build
```

Builds the production-ready application.

---

### Stage 2

```dockerfile
FROM node:20-alpine
```

Starts a new image using a lightweight Alpine Linux base image.

The previous build environment is discarded.

---

### Copy Build Output

```dockerfile
COPY --from=builder /app .
```

Copies the required files from the builder stage into the final runtime image.

General syntax:

```dockerfile
COPY --from=<stage-name> <source> <destination>
```

---

### Expose Port

```dockerfile
EXPOSE 3000
```

Makes the application available on port **3000**.

---

### Start Application

```dockerfile
CMD ["npm", "start"]
```

Starts the application when the container runs.

## GitHub Actions Workflow

```yaml
name: Multi-Stage Docker Build

on:
  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build Docker Image
        run: |
          docker build -t my-app:latest .
```

GitHub Actions automatically builds the Docker image using the multi-stage Dockerfile.

---

## Build and Push Workflow

```yaml
name: Build and Push Multi-Stage Image

on:
  workflow_dispatch:

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build Docker Image
        run: |
          docker build -t my-app:latest .

      - name: Tag Image
        run: |
          docker tag my-app:latest ${{ secrets.DOCKER_USERNAME }}/my-app:latest

      - name: Push Image
        run: |
          docker push ${{ secrets.DOCKER_USERNAME }}/my-app:latest
```

## Common Errors

### Build Stage Not Found

```text
COPY failed: invalid from flag value builder
```

**Solution**

Ensure the stage name matches:

```dockerfile
FROM node:20 AS builder
```

---

### Missing Build Files

```text
npm run build failed
```

**Solution**

Verify that:

- `package.json` exists.
- Build scripts are correctly defined.
- Dependencies install successfully.

---

### File Not Found

```text
COPY failed
```

**Solution**

Ensure the source path exists in the builder stage.

---

## Best Practices

✅ Use separate build and runtime stages

✅ Use lightweight runtime images (Alpine, Slim, Distroless)

✅ Copy only the required files into the final image

✅ Use Docker layer caching for faster builds

✅ Remove unnecessary build dependencies from the runtime image

❌ Don't include source code if it isn't needed at runtime

❌ Don't install development tools in the production image

❌ Don't use large base images unless necessary

## Simple Classroom Example

Imagine building a science project.

- **Builder Stage** = Workshop where you cut, paint, and assemble the project.
- **Runtime Stage** = Exhibition table where only the finished project is displayed.
- **Tools** = Scissors, glue, paint, and rulers.
- **Finished Model** = Final Docker image.

Visitors only need to see the completed project—not all the tools used to build it.

**In one line:** A Multi-Stage Docker Build creates smaller, faster, and more secure Docker images by separating the build process from the final runtime environment.
