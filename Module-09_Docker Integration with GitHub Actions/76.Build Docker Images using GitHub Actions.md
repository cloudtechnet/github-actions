# 71. Build Docker Images using GitHub Actions

### What is Building a Docker Image?

Building a Docker image means packaging your application along with all its dependencies, libraries, and runtime into a portable image that can run consistently on any system.

GitHub Actions can automatically build Docker images whenever code is pushed to the repository or when a workflow is triggered manually.

---

## Why Build Docker Images with GitHub Actions?

Without GitHub Actions, developers need to manually build Docker images on their local machines.

❌ Manual Process

```text
Developer Writes Code
        ↓
Open Terminal
        ↓
Run docker build
        ↓
Fix Build Errors
        ↓
Build Image Again
```

This process is time-consuming and inconsistent.

---

✅ Automated Process

```text
Developer Pushes Code
        ↓
GitHub Actions Starts
        ↓
Checks Out Repository
        ↓
Sets Up Docker Environment
        ↓
Builds Docker Image
        ↓
Build Completed Successfully
```

Every code change is built automatically, ensuring consistency.

---

## Prerequisites

Before building Docker images using GitHub Actions, ensure you have:

- A GitHub repository
- A Dockerfile in the project
- GitHub Actions enabled
- Docker Buildx (recommended for modern builds)

## Sample Dockerfile

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY . .

RUN pip install -r requirements.txt

CMD ["python", "app.py"]
```

---

## Step 1: Create a GitHub Actions Workflow

Create the following file:

```text
.github/workflows/docker-build.yml
```

---

## Step 2: Build Docker Image

```yaml
name: Build Docker Image

on:
  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build Docker Image
        run: docker build -t my-app:latest .
```

---

## Understanding the Workflow

### Workflow Name

```yaml
name: Build Docker Image
```

This gives the workflow a readable name in the GitHub Actions dashboard.

---

### Trigger

```yaml
on:
  workflow_dispatch:
```

This allows the workflow to be started manually from the GitHub Actions page.

---

### Job

```yaml
jobs:
  build:
```

Creates a job named **build**.

---

### Runner

```yaml
runs-on: ubuntu-latest
```

GitHub provides an Ubuntu virtual machine where the workflow executes.

---

### Checkout Repository

```yaml
- name: Checkout Repository
  uses: actions/checkout@v4
```

Downloads the repository source code into the GitHub runner.

Without this step, the workflow cannot access your project files or Dockerfile.

---

### Set Up Docker Buildx

```yaml
- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3
```

Configures Docker Buildx, which provides enhanced image-building capabilities such as:

- Faster builds
- Multi-platform image support
- Improved caching
- Advanced Docker build features

---

### Build Docker Image

```yaml
- name: Build Docker Image
  run: docker build -t my-app:latest .
```

Builds a Docker image using the Dockerfile in the current directory.

Here:

```text
docker build
```

Builds the image.

```text
-t my-app:latest
```

Assigns the image name **my-app** with the **latest** tag.

```text
.
```

Uses the current directory as the Docker build context.

## Build Image with a Custom Tag

Instead of using **latest**, you can specify a version tag.

```yaml
- name: Build Docker Image
  run: docker build -t my-app:v1.0 .
```

This creates:

```text
my-app:v1.0
```

---

## Build Using Git Commit SHA

A common practice is to tag images with the Git commit SHA.

```yaml
- name: Build Docker Image
  run: docker build -t my-app:${{ github.sha }} .
```

Example output:

```text
my-app:8d2c5f4b1e0a6f9d...
```

This makes every image uniquely identifiable.

## Common Build Options

### Build Without Cache

```yaml
run: docker build --no-cache -t my-app:latest .
```

Forces Docker to rebuild every layer.

---

### Specify Dockerfile

```yaml
run: docker build -f Dockerfile -t my-app:latest .
```

Uses a specific Dockerfile.

---

### Build from Another Directory

```yaml
run: docker build -t my-app:latest ./backend
```

Builds the image using the **backend** directory.

---

## Common Errors

### Dockerfile Not Found

```text
unable to prepare context:
unable to evaluate symlinks in Dockerfile
```

**Solution**

Ensure the Dockerfile exists in the specified build context.

---

### Build Failed

```text
COPY failed
```

**Solution**

Verify that all files referenced in the Dockerfile exist.

---

### Package Installation Failed

```text
RUN pip install ...
```

**Solution**

Check your dependency file (such as `requirements.txt`) and internet access during the build.

---

## Best Practices

✅ Keep Dockerfiles simple and optimized

✅ Use lightweight base images (such as Alpine or Slim variants)

✅ Tag images with version numbers instead of relying only on `latest`

✅ Use Docker Buildx for better performance

✅ Cache dependencies when possible to speed up builds

❌ Don't install unnecessary packages

❌ Don't include secrets in the Docker image

❌ Don't build images manually for every deployment
