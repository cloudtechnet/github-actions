# 81. Tagging Docker Images Dynamically

### What is Dynamic Docker Image Tagging?

**Dynamic Docker Image Tagging** is the process of automatically assigning unique tags to Docker images during the build process instead of using a fixed tag like `latest`.

The tag can be generated using information such as:

- Git commit SHA
- GitHub Release version
- Git tag
- Branch name
- Build number
- Date and time

This makes every Docker image uniquely identifiable and easier to manage.

---

## Why Use Dynamic Tags?

Using only the `latest` tag makes it difficult to identify which version of the application is deployed.

❌ Static Tag

```text
Developer Pushes Code
        ↓
Docker Image
        ↓
my-app:latest
        ↓
Old Image Gets Replaced
```

It becomes difficult to know which code version the image contains.

---

✅ Dynamic Tag

```text
Developer Pushes Code
        ↓
GitHub Actions Starts
        ↓
Generate Dynamic Tag
        ↓
Build Docker Image
        ↓
my-app:v1.2.0
```

Every image has a unique and meaningful version.

---

## Benefits of Dynamic Tagging

- Easy version tracking
- Better rollback support
- Improved deployment management
- Prevents accidental overwriting
- Supports CI/CD automation

## GitHub Context Variables

GitHub Actions provides built-in variables that can be used for dynamic tagging.

| Variable | Description |
|-----------|-------------|
| `${{ github.sha }}` | Full Git commit SHA |
| `${{ github.ref_name }}` | Branch or tag name |
| `${{ github.run_number }}` | Workflow run number |
| `${{ github.repository }}` | Repository name |
| `${{ github.actor }}` | GitHub username |

---

## Tag Using Git Commit SHA

```yaml
- name: Build Docker Image
  run: |
    docker build -t my-app:${{ github.sha }} .
```

Example image:

```text
my-app:8d3a7c6b5f4e...
```

Each commit creates a unique Docker image.

---

## Tag Using Branch Name

```yaml
- name: Build Docker Image
  run: |
    docker build -t my-app:${{ github.ref_name }} .
```

Example:

```text
my-app:main

my-app:development

my-app:feature-login
```

---

## Tag Using Workflow Run Number

```yaml
- name: Build Docker Image
  run: |
    docker build -t my-app:${{ github.run_number }} .
```

Example:

```text
my-app:105
```

Each workflow execution generates a new tag.

---

## Tag Using GitHub Release Version

Suppose a GitHub Release is created:

```text
v1.5.0
```

Use:

```yaml
- name: Build Docker Image
  run: |
    docker build -t my-app:${{ github.ref_name }} .
```

Generated image:

```text
my-app:v1.5.0
```

---

## Build with Multiple Tags

It is common to build one image and assign multiple tags.

```yaml
- name: Build Docker Image
  run: |
    docker build \
      -t my-app:latest \
      -t my-app:${{ github.sha }} \
      -t my-app:${{ github.run_number }} .
```

Generated images:

```text
my-app:latest

my-app:8d3a7c6

my-app:105
```

## Complete GitHub Actions Workflow

```yaml
name: Dynamic Docker Image Tagging

on:
  workflow_dispatch:

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build Docker Image
        run: |
          docker build \
            -t my-app:latest \
            -t my-app:${{ github.sha }} \
            -t my-app:${{ github.run_number }} .
```

---

## Build and Push Dynamically Tagged Images

```yaml
name: Build and Push Docker Image

on:
  workflow_dispatch:

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build Image
        run: |
          docker build \
            -t my-app:latest \
            -t my-app:${{ github.sha }} .

      - name: Tag Image
        run: |
          docker tag my-app:latest ${{ secrets.DOCKER_USERNAME }}/my-app:latest
          docker tag my-app:${{ github.sha }} ${{ secrets.DOCKER_USERNAME }}/my-app:${{ github.sha }}

      - name: Push Images
        run: |
          docker push ${{ secrets.DOCKER_USERNAME }}/my-app:latest
          docker push ${{ secrets.DOCKER_USERNAME }}/my-app:${{ github.sha }}
```

## Common Errors

### Invalid Tag Format

```text
invalid reference format
```

**Solution**

Ensure the generated tag contains only valid Docker tag characters.

---

### Tag Not Found

```text
No such image
```

**Solution**

Verify that the image was built successfully before applying additional tags.

---

### Push Failed

```text
requested access to the resource is denied
```

**Solution**

Ensure you are logged in to the container registry and have permission to push images.

---

## Best Practices

✅ Tag every image with a unique version

✅ Use both `latest` and a version-specific tag

✅ Use Git commit SHA for traceability

✅ Use GitHub Releases for production versions

✅ Keep image tags consistent across environments

❌ Don't rely only on the `latest` tag

❌ Don't overwrite production images unintentionally

❌ Don't use random or meaningless tag names

## Simple Classroom Example

Imagine submitting school assignments.

- **Assignment** = Docker Image
- **Roll Number** = Git Commit SHA
- **Submission Number** = Workflow Run Number
- **Final Exam Version** = GitHub Release Tag
- **Teacher** = Deployment System

Instead of submitting every assignment with the same name, each one receives a unique identifier so the teacher can easily identify which version was submitted.

**In one line:** Dynamic Docker image tagging automatically assigns meaningful and unique tags—such as commit SHAs, release versions, or build numbers—to Docker images, making them easier to identify, deploy, and roll back in CI/CD pipelines.
