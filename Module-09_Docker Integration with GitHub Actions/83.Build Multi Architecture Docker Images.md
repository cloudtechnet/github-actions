# 83. Build Multi-Architecture Docker Images

### What are Multi-Architecture Docker Images?

**Multi-Architecture Docker Images** are Docker images that support multiple CPU architectures, allowing the same image to run on different hardware platforms.

Common architectures include:

- **linux/amd64** (Intel and AMD 64-bit processors)
- **linux/arm64** (Apple Silicon, AWS Graviton, Raspberry Pi 4)
- **linux/arm/v7** (Older Raspberry Pi devices)

Using **Docker Buildx**, GitHub Actions can automatically build and publish a single Docker image that works across multiple architectures.

---

## Why Build Multi-Architecture Images?

Without multi-architecture support, separate Docker images must be built for each platform.

❌ Single Architecture Build

```text
Developer Builds Image
        ↓
linux/amd64 Image
        ↓
Runs Only on Intel/AMD Systems
```

Users with ARM-based devices cannot run the image.

---

✅ Multi-Architecture Build

```text
Developer Pushes Code
        ↓
GitHub Actions Starts
        ↓
Docker Buildx
        ↓
Build amd64 Image
        ↓
Build arm64 Image
        ↓
Create Multi-Architecture Image
        ↓
Push to Registry
```

A single image automatically supports multiple platforms.

---

## Benefits of Multi-Architecture Images

- One image supports multiple CPU architectures
- Works on Intel, AMD, Apple Silicon, and ARM devices
- Simplifies deployments
- Improves portability
- Reduces maintenance effort
- Ideal for cloud, edge, and IoT environments

---

## Common CPU Architectures

| Architecture | Common Devices |
|--------------|----------------|
| linux/amd64 | Intel and AMD PCs, servers, cloud VMs |
| linux/arm64 | Apple Silicon Macs, AWS Graviton, Azure ARM VMs |
| linux/arm/v7 | Raspberry Pi (32-bit), embedded devices |

## Prerequisites

Before building multi-architecture images, ensure you have:

- GitHub repository
- Dockerfile
- GitHub Actions enabled
- Docker Buildx
- Docker Hub or GitHub Container Registry account

---

## Step 1: Checkout Repository

```yaml
- name: Checkout Repository
  uses: actions/checkout@v4
```

---

## Step 2: Set Up QEMU

QEMU enables emulation for different CPU architectures during the build.

```yaml
- name: Set up QEMU
  uses: docker/setup-qemu-action@v3
```

---

## Step 3: Set Up Docker Buildx

```yaml
- name: Set up Docker Buildx
  uses: docker/setup-buildx-action@v3
```

Buildx provides multi-platform build capabilities.

---

## Step 4: Login to Docker Hub

```yaml
- name: Login to Docker Hub
  uses: docker/login-action@v3
  with:
    username: ${{ secrets.DOCKER_USERNAME }}
    password: ${{ secrets.DOCKER_PASSWORD }}
```

---

## Step 5: Build and Push Multi-Architecture Image

```yaml
- name: Build and Push Docker Image
  uses: docker/build-push-action@v6
  with:
    context: .
    push: true
    tags: ${{ secrets.DOCKER_USERNAME }}/my-app:latest
    platforms: linux/amd64,linux/arm64
```

This builds images for both architectures and publishes a single multi-architecture image.

---

## Complete GitHub Actions Workflow

```yaml
name: Build Multi-Architecture Docker Images

on:
  workflow_dispatch:

jobs:
  docker:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build and Push Docker Image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ secrets.DOCKER_USERNAME }}/my-app:latest
          platforms: linux/amd64,linux/arm64
```

## Understanding the Workflow

### Set Up QEMU

```yaml
uses: docker/setup-qemu-action@v3
```

Enables CPU emulation, allowing builds for architectures different from the GitHub runner's architecture.

---

### Set Up Docker Buildx

```yaml
uses: docker/setup-buildx-action@v3
```

Creates a Buildx builder capable of building multi-platform Docker images.

---

### Login to Docker Hub

```yaml
uses: docker/login-action@v3
```

Authenticates with Docker Hub using GitHub Secrets.

---

### Build and Push

```yaml
uses: docker/build-push-action@v6
```

Builds and pushes the Docker image in one step.

---

### Build Context

```yaml
context: .
```

Uses the current directory as the Docker build context.

---

### Push

```yaml
push: true
```

Automatically pushes the image after a successful build.

---

### Tags

```yaml
tags: username/my-app:latest
```

Specifies the Docker image name and tag.

---

### Platforms

```yaml
platforms: linux/amd64,linux/arm64
```

Builds the image for both Intel/AMD and ARM64 architectures.

---

## Image Manifest

After the push, Docker creates a **multi-architecture manifest**.

Example:

```text
my-app:latest

├── linux/amd64
└── linux/arm64
```

When a user runs:

```bash
docker pull username/my-app:latest
```

Docker automatically downloads the correct image for the user's system.

---

## Build Multiple Platforms

You can build for additional platforms.

```yaml
platforms: |
  linux/amd64,
  linux/arm64,
  linux/arm/v7
```

Supported devices include:

- Intel/AMD servers
- Apple Silicon Macs
- AWS Graviton
- Raspberry Pi
- Embedded Linux devices

## Common Errors

### Buildx Not Found

```text
docker: buildx is not available
```

**Solution**

Use:

```yaml
uses: docker/setup-buildx-action@v3
```

before building.

---

### Unsupported Platform

```text
no match for platform
```

**Solution**

Ensure the requested architecture is supported by the base image.

---

### Authentication Failed

```text
unauthorized
```

**Solution**

Verify Docker registry credentials stored in GitHub Secrets.

---

## Best Practices

✅ Use Docker Buildx for all multi-platform builds

✅ Enable QEMU before building

✅ Publish a single multi-architecture image instead of separate images

✅ Test images on supported architectures

✅ Use lightweight base images when possible

❌ Don't maintain separate Dockerfiles for different architectures unless required

❌ Don't assume every base image supports all architectures

❌ Don't skip testing on ARM-based systems
