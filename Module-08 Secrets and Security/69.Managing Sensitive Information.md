# Managing Sensitive Information in GitHub Actions

**Managing Sensitive Information** means storing and using confidential data securely in your CI/CD pipelines without exposing it in code, logs, or repositories.

Examples of sensitive information:

* Passwords
* API Keys
* Access Tokens
* Database Credentials
* SSH Keys
* Cloud Credentials (AWS, Azure, GCP)
* Docker Hub Credentials

## Why is it Important?

Imagine you have this code:

```yaml
env:
  AWS_ACCESS_KEY_ID: ABCDEFG12345
  AWS_SECRET_ACCESS_KEY: xyz123456789
```

If this file is pushed to GitHub:

* Anyone with repository access can see it.
* Credentials may get leaked.
* Attackers can access your cloud resources.
* Security risks increase.

To avoid this, GitHub provides **Secrets**.

---

## Real-World Example

Suppose your workflow needs to deploy an application to AWS.

The workflow needs:

* AWS Access Key
* AWS Secret Key

Instead of hardcoding them:

```yaml
AWS_ACCESS_KEY_ID: ABCDEFG12345
AWS_SECRET_ACCESS_KEY: xyz123456789
```

Store them in GitHub Secrets:

```text
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

Then access them securely:

```yaml
env:
  AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
  AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

Now the actual values are hidden and protected.

## GitHub Secrets

GitHub Secrets are encrypted values stored securely inside GitHub.

Example:

```text
DOCKER_USERNAME
DOCKER_PASSWORD
AWS_ACCESS_KEY_ID
AWS_SECRET_ACCESS_KEY
```

GitHub encrypts these values and only exposes them during workflow execution.

---

## How Secrets Work

### Step 1

Store Secret in GitHub:

```text
Settings
 → Secrets and Variables
 → Actions
 → New Repository Secret
```

Add:

```text
Name: DOCKER_PASSWORD
Value: MySecurePassword
```

### Step 2

Use Secret in Workflow:

```yaml
steps:
  - name: Login to DockerHub
    run: |
      docker login -u ${{ secrets.DOCKER_USERNAME }} \
      -p ${{ secrets.DOCKER_PASSWORD }}
```

---

## What Happens During Execution?

Workflow:

```yaml
echo ${{ secrets.API_KEY }}
```

Log Output:

```text
***
```

GitHub automatically masks secret values in logs.

This prevents accidental exposure.

---

## Best Practices

### ✅ Use Secrets

Store sensitive data in GitHub Secrets.

### ✅ Rotate Credentials Regularly

Change passwords and tokens periodically.

### ✅ Use Least Privilege

Give only required permissions.

Example:

```text
Read Only Access
```

instead of

```text
Administrator Access
```

### ✅ Avoid Hardcoding

Never do this:

```yaml
password: admin123
```

### ✅ Review Workflow Logs

Ensure secrets are not accidentally printed.

## Student-Friendly Analogy

Think of a **GitHub Secret** like an **ATM PIN**.

* ATM Card = GitHub Workflow
* PIN = Secret
* Bank Locker = GitHub Secrets

You never write your ATM PIN on the card.

Similarly, you should never write passwords or API keys directly in workflow files.

Store them securely in GitHub Secrets and use them only when needed.

## One-Line Definition for Interviews

**Managing Sensitive Information is the practice of securely storing and using confidential data such as passwords, API keys, tokens, and cloud credentials in GitHub Actions using Secrets, without exposing them in code, repositories, or logs.**
