# 22. Default Environment Variables

GitHub Actions automatically provides several built-in environment variables during workflow execution.

These are called:

> Default Environment Variables

They contain useful information about:

* Repository
* Branch
* Commit
* Workflow
* Runner
* Event details
* User information

You can use them directly in workflows without defining them manually.

---

# Why Default Variables are Important

Default variables help workflows become:

* Dynamic
* Reusable
* Intelligent
* Context-aware

Instead of hardcoding values, GitHub provides information automatically.

---

# Example

Without variables:

```yaml
run: echo "Deploying from main branch"
```

Hardcoded.

Using GitHub variable:

```yaml
run: echo "Deploying from $GITHUB_REF"
```

Dynamic.

---

# Types of Default Variables

GitHub provides variables related to:

| Category   | Examples            |
| ---------- | ------------------- |
| Repository | `GITHUB_REPOSITORY` |
| Branch/Tag | `GITHUB_REF`        |
| Commit     | `GITHUB_SHA`        |
| User       | `GITHUB_ACTOR`      |
| Workflow   | `GITHUB_WORKFLOW`   |
| Runner     | `RUNNER_OS`         |
| Event      | `GITHUB_EVENT_NAME` |

---

# How to Access Default Variables

Inside shell commands:

```yaml
run: echo $GITHUB_REPOSITORY
```

Inside expressions:

```yaml
${{ github.repository }}
```

---

# Important Difference

| Method               | Usage                      |
| -------------------- | -------------------------- |
| Environment Variable | `$GITHUB_REPOSITORY`       |
| GitHub Context       | `${{ github.repository }}` |

Both refer to same information.

---

# 1. Repository Variables

---

# GITHUB_REPOSITORY

Returns:

```text
owner/repository-name
```

---

# Example

```yaml
name: Repository Info

on: push

jobs:
  repo-info:
    runs-on: ubuntu-latest

    steps:
      - name: Show Repository
        run: echo "Repository: $GITHUB_REPOSITORY"
```

---

# Sample Output

```text
Repository: cloudtech/demo-project
```

---

# GITHUB_REPOSITORY_OWNER

Returns repository owner.

---

# Example

```yaml
run: echo $GITHUB_REPOSITORY_OWNER
```

Output:

```text
cloudtech
```

---



| Branch/Tag  | Output             |

Returns short branch/tag name.

# Example


Output:

```text
main
```

---

# Difference

| Variable          | Output          |
| ----------------- | --------------- |
| `GITHUB_REF`      | refs/heads/main |

# 12. Using GitHub Context Instead of Variables

GitHub also provides contexts.

---

# Example

```yaml
run: echo "${{ github.repository }}"
```

Equivalent to:

```yaml
run: echo $GITHUB_REPOSITORY
```

---

# Common GitHub Contexts

| Context | Purpose               |
| ------- | --------------------- |
| github  | Workflow/event data   |
| env     | Environment variables |
| secrets | Secret values         |
| vars    | Repository variables  |
| runner  | Runner information    |
| matrix  | Matrix build data     |

---

# Example

```yaml
run: echo "${{ github.actor }}"
```

# 13. Real-World CI/CD Example

---

# Scenario

Deploy app using branch and commit details.

---

# Workflow

```yaml
name: Deployment Pipeline

on:
  push:
    branches:
      - main

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Build Info
        run: |
          echo "Deploying from branch: $GITHUB_REF_NAME"
          echo "Commit: $GITHUB_SHA"
          echo "Triggered by: $GITHUB_ACTOR"

      - name: Deploy
        run: |
          echo "Deployment successful"
```

# 14. Security Notes

Some variables contain sensitive information.

Avoid printing:

* Tokens
* Secrets
* Sensitive payloads

---

# Example Risk

```yaml
cat $GITHUB_EVENT_PATH
```

May expose sensitive event data.

---

# 15. Best Practices

| Best Practice                   | Reason               |
| ------------------------------- | -------------------- |
| Use default variables           | Avoid hardcoding     |
| Use `GITHUB_SHA` for versioning | Unique build IDs     |
| Use `GITHUB_REF_NAME`           | Cleaner branch names |
| Avoid exposing sensitive info   | Security             |
| Use contexts in conditions      | Better readability   |
