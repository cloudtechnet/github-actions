# 16. Workflow Events and Triggers in GitHub Actions

In GitHub Actions, **Workflow Events and Triggers** define **when a workflow should start automatically**.

A workflow runs only when a specific event happens in the repository.

Examples:

* Push code
* Create Pull Request
* Create Release
* Schedule jobs
* Manual execution
* Issue creation
* Tag creation

---

# 1️⃣ What is a Workflow Trigger?

A **trigger** is defined using the `on:` keyword inside the workflow YAML file.

## Basic Syntax

```yaml
name: My Workflow

on:
  push:
    branches:
      - main
```

This workflow runs whenever code is pushed to the `main` branch.

---

# 2️⃣ Common Workflow Events

| Event             | Purpose                           |
| ----------------- | --------------------------------- |
| push              | Runs when code is pushed          |
| pull_request      | Runs during PR creation/update    |
| workflow_dispatch | Manual trigger                    |
| schedule          | Run automatically on time         |
| release           | Trigger on GitHub releases        |
| issues            | Trigger when issue created/closed |
| fork              | Trigger when repo forked          |
| create            | Trigger when branch/tag created   |
| delete            | Trigger when branch/tag deleted   |
| watch             | Trigger when someone stars repo   |

# 3️⃣ Push Event
#  Push and Pull Request Events

In GitHub Actions, an **event** is something that triggers a workflow automatically.

Two of the most commonly used events are:

1. `push`
2. `pull_request`

These events help automate:

* Testing
* Building
* Deployment
* Code validation
* CI/CD pipelines

---

# 1. Push Event

A `push` event triggers a workflow whenever code is pushed to a repository.

## Syntax

```yaml
on:
  push:
```

---

# How Push Event Works

When developer pushes code:

```bash
git push origin main
```

GitHub detects the push and automatically starts the workflow.

---

# Basic Push Event Example

```yaml
name: Push Workflow

on:
  push:

jobs:
  demo:
    runs-on: ubuntu-latest

    steps:
      - name: Print Message
        run: echo "Code pushed successfully!"
```

---

# Explanation

| Section    | Description              |
| ---------- | ------------------------ |
| `on: push` | Trigger workflow on push |
| `jobs`     | Group of tasks           |
| `runs-on`  | Operating system         |
| `steps`    | Commands/actions         |

---

# Real-Time Scenario

Developer pushes new code:

```bash
git add .
git commit -m "Added login feature"
git push origin main
```

GitHub Actions automatically:

1. Starts workflow
2. Executes jobs
3. Shows result in Actions tab

---

# Push Event on Specific Branch

Sometimes workflows should run only on certain branches.

Example:

```yaml
on:
  push:
    branches:
      - main
```

Workflow runs only when code is pushed to:

```text
main
```

---

# Multiple Branch Example

```yaml
on:
  push:
    branches:
      - main
      - develop
```

Now workflow triggers for both:

* `main`
* `develop`

---

# Ignore Branch Example

```yaml
on:
  push:
    branches-ignore:
      - experimental
```

Workflow will not run for:

```text
experimental
```

---

# Push Event With Tags

```yaml
on:
  push:
    tags:
      - 'v*'
```

Triggers when tags like below are pushed:

```text
v1.0
v2.0
```

---

# Real Project Example – Automated Testing

```yaml

  push:
    branches:
      - main

  testing:
    runs-on: ubuntu-latest

      - name: Checkout Code
```

# 5️⃣ Manual Trigger — workflow_dispatch

`workflow_dispatch` is a **manual trigger event** in GitHub Actions.
It allows you to start a workflow manually from:

* GitHub UI (Actions tab)
* GitHub CLI
* API
* External tools/scripts

This is useful when you want to run workflows **on demand** instead of automatically on every push or pull request.

---

# Why Use `workflow_dispatch`?

Normally workflows run automatically like:

* `push`
* `pull_request`
* `schedule`

But sometimes you need manual control.

Examples:

* Deploy application manually
* Run backup process
* Trigger production release
* Execute testing only when needed
* Select environment dynamically
* Run workflow with custom inputs

---

# Basic Syntax

```yaml
on:
  workflow_dispatch:
```

Example:

```yaml
name: Manual Workflow

on:
  workflow_dispatch:

jobs:
  demo:
    runs-on: ubuntu-latest

    steps:
      - name: Print Message
        run: echo "Workflow triggered manually!"
```

---

# How It Works

After pushing this workflow file:

1. Open GitHub repository
2. Go to **Actions**
3. Select workflow
4. Click **Run workflow**
5. Choose branch
6. Click **Run workflow** button

Now GitHub starts the workflow manually.

---

# Real-Time Example

## Example: Manual Deployment

```yaml
name: Deploy Application

on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Deploy App
        run: echo "Deploying application..."
```

Use case:

* Developer tests code
* Team lead manually deploys production
* Avoids automatic production deployments

---

# Workflow Dispatch With Inputs

You can ask users for values before running workflow.

Example:

```yaml
name: Environment Deployment

on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Select Environment'
        required: true
        default: 'dev'

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Show Environment
        run: echo "Deploying to ${{ github.event.inputs.environment }}"
```

---

# What Happens Here?

Before running workflow GitHub shows:

| Input       | Value             |
| ----------- | ----------------- |
| environment | dev / test / prod |

User selects value manually.

---

# Example Output

If user selects:

```text
prod
```

Output:

```text
Deploying to prod
```

---

# 6️⃣ Scheduled Workflow (Cron Jobs)

In GitHub Actions, the `schedule` event is used to run workflows automatically at specific times using **Cron expressions**.

This is useful for:

* Daily backups
* Automated testing
* Security scans
* Cleanup jobs
* Report generation
* Nightly builds

---

# What is a Schedule Trigger?

A schedule trigger automatically starts a workflow at a fixed time interval.

GitHub uses:

```text
Cron Syntax
```

to define the schedule.

---

# Basic Syntax

```yaml id="ec12nt"
on:
  schedule:
    - cron: '0 0 * * *'
```

---

# What is Cron?

A cron expression defines:

```text
When a workflow should run
```

Cron format contains 5 fields:

```text
* * * * *
│ │ │ │ │
│ │ │ │ └── Day of Week (0-6)
│ │ │ └──── Month (1-12)
│ │ └────── Day of Month (1-31)
│ └──────── Hour (0-23)
└────────── Minute (0-59)
```

---

# Cron Field Explanation

| Field        | Range | Example      |
| ------------ | ----- | ------------ |
| Minute       | 0–59  | `30`         |
| Hour         | 0–23  | `10`         |
| Day of Month | 1–31  | `15`         |
| Month        | 1–12  | `5`          |
| Day of Week  | 0–6   | `1` = Monday |

---

# Important Note

GitHub Actions schedule time uses:

```text
UTC Time Zone
```

not local time.

---

# Basic Example

## Run Workflow Every Day at Midnight

```yaml id="6gqlt3"
name: Daily Workflow

on:
  schedule:
    - cron: '0 0 * * *'

jobs:
  daily-job:
    runs-on: ubuntu-latest

    steps:
      - name: Print Message
        run: echo "Daily workflow executed!"
```

---

# Explanation

```text
0 0 * * *
```

Means:

| Value | Meaning       |
| ----- | ------------- |
| 0     | Minute        |
| 0     | Hour          |
| *     | Every day     |
| *     | Every month   |
| *     | Every weekday |

Result:

```text
Runs every day at 12:00 AM UTC
```

---

# 8️⃣ Branch Filtering

#  Filtering Branches and Paths

In GitHub Actions, **Filtering Branches and Paths** helps you control **when a workflow should run**.

Instead of running workflows for every push or pull request, you can tell GitHub Actions:

* Run only for specific branches
* Run only when specific files/folders change
* Ignore some branches or files

This improves:

* Performance
* Faster CI/CD
* Reduced unnecessary workflow runs
* Better project organization

---

# 1. Branch Filtering

Branch filtering means:

> Trigger workflows only for selected branches.

---

# Why Use Branch Filters?

Example:

You may have branches like:

* `main`
* `develop`
* `feature/login`
* `feature/payment`

You may want:

* Deploy only from `main`
* Run testing on `develop`
* Ignore temporary branches

---

# Syntax

```yaml
on:
  push:
    branches:
      - main
```

This workflow runs only when code is pushed to the `main` branch.

---

# Example 1 – Run Workflow Only on Main Branch

```yaml
name: Main Branch Workflow

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Display Message
        run: echo "Code pushed to main branch"
```

---

# How It Works

If you push to:

| Branch        | Workflow Runs? |
| ------------- | -------------- |
| main          | ✅ Yes          |
| develop       | ❌ No           |
| feature/login | ❌ No           |

---

# Example 2 – Multiple Branches

```yaml
on:
  push:
    branches:
      - main
      - develop
```

Runs workflow on:

* `main`
* `develop`

---

# Example 3 – Wildcard Branch Matching

```yaml
on:
  push:
    branches:
      - 'feature/*'
```

Matches:

* `feature/login`
* `feature/payment`

Does NOT match:

* `bugfix/error`

---

# Wildcard Patterns

| Pattern     | Meaning                           |
| ----------- | --------------------------------- |
| `feature/*` | Any branch starting with feature/ |
| `release/*` | Any release branch                |
| `hotfix/*`  | Any hotfix branch                 |
| `**`        | Matches all branches              |

---

# Example 4 – Ignore Branches

Use `branches-ignore`.

```yaml
on:
  push:
    branches-ignore:
      - develop
```

Runs on all branches EXCEPT:

* `develop`

---

# Example 5 – Branch Filter with Pull Requests

```yaml
on:
  pull_request:
    branches:
      - main
```

Workflow runs when:

* Pull request targets `main`

Example:

```text
feature/login  --->  main
```

Workflow runs because target branch is `main`.

# 2. Path Filtering

Path filtering means:

> Run workflows only when specific files or folders change.

---

# Why Use Path Filters?

Suppose your repository contains:

```text
project/
 ├── frontend/
 ├── backend/
 ├── docs/
```

You may want:

* Frontend workflow only if frontend files change
* Backend workflow only if backend files change
* Ignore documentation changes

---

# Syntax

```yaml
on:
  push:
    paths:
      - 'frontend/**'
```

Workflow runs only if files inside `frontend` change.

---

# Example 1 – Frontend Changes Only

```yaml
name: Frontend Workflow

on:
  push:
    paths:
      - 'frontend/**'

jobs:
  frontend-build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - run: echo "Frontend files changed"
```

# How It Works

# 5. Advanced Wildcards

| Pattern | Meaning                          |
| ------- | -------------------------------- |
| `*`     | Matches any character except `/` |
| `**`    | Matches multiple directories     |
| `?.js`  | Single character match           |
| `[abc]` | Match a, b, or c                 |

---

# Examples

| Pattern     | Matches                |
| ----------- | ---------------------- |
| `src/*.js`  | src/app.js             |
| `src/**`    | All files under src    |
| `**/*.yml`  | All YAML files         |
| `docs/*.md` | Markdown files in docs |
