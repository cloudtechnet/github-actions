# 27. Timeout and Retry Configuration

In GitHub Actions, workflows sometimes:

* Take too long
* Get stuck
* Fail because of temporary issues
* Face network/API problems

To handle these situations, GitHub Actions provides:

1. **Timeout Configuration**
2. **Retry Mechanisms**

These help create:

* Reliable CI/CD pipelines
* Stable automation
* Faster failure handling
* Production-ready workflows

---

# 1. What is Timeout Configuration?

A timeout defines:

> Maximum time allowed for a workflow job or step to run.

If execution exceeds the limit:

* GitHub automatically stops it
* Job becomes failed

---

# Why Timeouts are Important

Without timeout:

* Jobs may hang forever
* Runners remain occupied
* CI resources get wasted
* Pipelines become slow

---

# Real Example

Suppose:

* Build process freezes
* Test server never responds
* Docker pull hangs

Without timeout:

```text id="5w8u1j"
Workflow keeps running forever
```

With timeout:

```text id="u4m0z9"
Workflow stops automatically after defined time
```

---

# Default Timeout in GitHub Actions

GitHub automatically applies default limits.

| Runner Type          | Default Timeout       |
| -------------------- | --------------------- |
| GitHub-hosted runner | 360 minutes (6 hours) |
| Self-hosted runner   | Can vary              |

---

# Maximum Timeout

| Runner        | Maximum             |
| ------------- | ------------------- |
| GitHub-hosted | 360 minutes         |
| Self-hosted   | Higher/customizable |

---

# 2. Job-Level Timeout

Use:

```yaml id="2o9rbj"
timeout-minutes:
```

---

# Syntax

```yaml id="ihxk8j"
jobs:
  build:
    timeout-minutes: 10
```

Meaning:

> Stop job if it runs more than 10 minutes.

---

# Example – Basic Timeout

```yaml id="jlwmj0"
name: Timeout Example

on: push

jobs:
  build:
    runs-on: ubuntu-latest

    timeout-minutes: 5

    steps:
      - name: Simulate Long Task
        run: sleep 600
```

---

# What Happens?

* `sleep 600` = 10 minutes
* Timeout = 5 minutes

Result:

```text id="a3a6o1"
Job cancelled after 5 minutes
```

---

# Execution Flow

```text id="e46x7p"
Job Starts
    ↓
Runs for 5 Minutes
    ↓
GitHub Stops Job Automatically
```

---

# 3. Timeout in Real CI/CD

---

# Example Scenario

| Task           | Suggested Timeout |
| -------------- | ----------------- |
| Unit tests     | 10 min            |
| Docker build   | 20 min            |
| Deployment     | 15 min            |
| Security scans | 30 min            |

---

# Example


# 4. Step-Level Timeout

GitHub Actions does NOT directly support:

```yaml id="w2j30h"
step timeout-minutes
```

for normal `run` steps.

But some actions support their own timeout options.

---

# Example with Third-Party Action

```yaml id="3fjlwm"
- uses: some-action@v1
  timeout-minutes: 5
```

Depends on action support.

---

# Alternative Solution

Use Linux timeout command.

---

# Example

```yaml id="o7rlbn"
steps:
  - name: Timeout Command
    run: timeout 60s ping google.com
```

Meaning:

* Stop command after 60 seconds

---

# 5. Timeout with Matrix Jobs

Each matrix job gets its own timeout.

---

# Example

```yaml id="qvzc6q"
jobs:
  test:
    strategy:
      matrix:
        node: [16, 18, 20]

    runs-on: ubuntu-latest

    timeout-minutes: 15

    steps:
      - run: echo "Testing Node ${{ matrix.node }}"
```

---

# Result

Each parallel job:

* Has separate 15-minute timeout

# 6. What is Retry Configuration?

Retry means:

> Automatically re-running failed operations.

Useful because many failures are temporary:

* Network issues
* API timeouts
* Package download failures
* Docker registry failures

---

# Why Retry is Important

Without retry:

```text id="wyvmuy"
Temporary network failure → Workflow fails
```

With retry:

```text id="f3xmh8"
Temporary failure → Retry → Success
```

---

# Important Note

GitHub Actions has:

❌ No built-in automatic retry for jobs/steps

But retries can be implemented using:

* Shell scripting
* Third-party retry actions
* Loops
* Workflow reruns

---

# 7. Simple Retry Using Shell Script

---

# Example

```yaml id="8o3e8e"
steps:
  - name: Retry Example
    run: |
      for i in 1 2 3
      do
        curl https://example.com && break
        echo "Retrying..."
        sleep 5
      done
```

---

# How It Works

| Attempt | Action         |
| ------- | -------------- |
| 1       | Try request    |
| Fail    | Wait 5 seconds |
| 2       | Retry          |
| Fail    | Retry again    |
| 3       | Final attempt  |

---

# 8. Retry with Until Loop

---

# Example

```yaml id="y5w0eb"
steps:
  - name: Retry Until Success
    run: |
      count=0

      until curl https://example.com
      do
        count=$((count+1))

        if [ $count -ge 5 ]; then
          echo "Failed after 5 attempts"
          exit 1
        fi

        echo "Retrying..."
        sleep 10
      done
```

---

# Logic

```text id="cqv7ef"
Try Request
   ↓
Success? ──► YES → Exit
   ↓ NO
Retry
   ↓
Max Attempts Reached?
   ↓
Fail Workflow
```

---

# 9. Retry Using GitHub Marketplace Actions

GitHub Marketplace provides retry actions.

Example:

* retry-step
* retry-action

---

# Example

```yaml id="e0piyf"
- name: Retry API Call
  uses: nick-fields/retry@v3

  with:
    timeout_minutes: 1
    max_attempts: 3
    command: curl https://example.com
```

---

# Features

| Option            | Meaning             |
| ----------------- | ------------------- |
| `max_attempts`    | Retry count         |
| `timeout_minutes` | Timeout per attempt |
| `command`         | Command to retry    |

---

# 10. Retry Failed Jobs Manually

GitHub UI allows rerunning jobs.

---

# Options

| Option             | Purpose             |
| ------------------ | ------------------- |
| Re-run all jobs    | Restart workflow    |
| Re-run failed jobs | Retry only failures |

---

# Useful For

* Temporary failures
* External service outages
* Network instability

---

# 11. Combining Timeout and Retry

Best practice in enterprise CI/CD.

---

# Example

```yaml id="o41tgg"
name: Retry and Timeout

on: push

jobs:
  api-test:
    runs-on: ubuntu-latest

    timeout-minutes: 10

    steps:
      - name: Retry API Test
        run: |
          for i in 1 2 3
          do
            curl https://example.com && break

            echo "Attempt $i failed"

            sleep 5
          done
```

---

# Benefits

| Feature | Benefit                    |
| ------- | -------------------------- |
| Timeout | Prevent hanging            |
| Retry   | Recover temporary failures |

# 12. Real-World CI/CD Example

---

# Scenario

Pipeline includes:

* Docker build
* Dependency install
* API testing
* Deployment

Some operations may fail temporarily.

---

# Workflow

```yaml id="z7g53n"
name: Enterprise Pipeline

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    timeout-minutes: 20

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Install Dependencies with Retry
        run: |
          for i in 1 2 3
          do
            npm install && break

            echo "npm install failed"
            sleep 10
          done

      - name: Build Application
        run: npm run build

      - name: API Test with Retry
        run: |
          count=0

          until curl https://example.com/api/health
          do
            count=$((count+1))

            if [ $count -ge 5 ]; then
              exit 1
            fi

            sleep 5
          done
```

---

# Pipeline Features

| Feature          | Purpose                      |
| ---------------- | ---------------------------- |
| Job timeout      | Avoid long hangs             |
| Retry install    | Handle npm failures          |
| Retry API checks | Handle server startup delays |

---

# 13. Timeout vs Retry

| Feature       | Timeout             | Retry              |
| ------------- | ------------------- | ------------------ |
| Purpose       | Stop long execution | Reattempt failures |
| Trigger       | Time exceeded       | Command failure    |
| Prevents      | Hanging workflows   | Temporary failures |
| Configuration | `timeout-minutes`   | Scripts/actions    |

---

# 14. Best Practices

| Best Practice                 | Reason               |
| ----------------------------- | -------------------- |
| Always set timeouts           | Prevent hanging jobs |
| Use retries for network calls | Improve reliability  |
| Keep retry count limited      | Avoid infinite loops |
| Add delay between retries     | Reduce server load   |
| Combine retry + timeout       | Stable pipelines     |

# 15. Common Mistakes

---

# Mistake 1 – No Timeout

Problem:

```text id="9swhf0"
Workflow hangs forever
```

---

# Mistake 2 – Infinite Retry Loops

Bad example:

```yaml id="s4as5u"
while true
```

Can run forever.

---

# Mistake 3 – Too Many Retries

Excessive retries:

* Waste CI minutes
* Slow pipelines

---

# Mistake 4 – Retrying Permanent Errors

Retry won't fix:

* Syntax errors
* Missing files
* Invalid credentials

# 16. Advanced Example

```yaml id="x1ghwq"
name: Advanced Retry and Timeout

on: push

jobs:
  deploy:
    runs-on: ubuntu-latest

    timeout-minutes: 15

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Retry Deployment
        run: |
          attempts=0

          until ./deploy.sh
          do
            attempts=$((attempts+1))

            if [ $attempts -ge 3 ]; then
              echo "Deployment failed"
              exit 1
            fi

            echo "Retrying deployment..."
            sleep 10
          done

      - name: Verify Deployment
        run: echo "Deployment successful"
```
