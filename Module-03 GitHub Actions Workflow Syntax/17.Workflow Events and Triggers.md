# 17. Workflow Events and Triggers in GitHub Actions

In GitHub Actions, **Workflow Events and Triggers** define **when a workflow should start automatically**.

A workflow runs only when a specific event happens in the repository.

Examples:

* Push code
* Create Pull Request
* Create Release
* Schedule jobs
* Manual execution
* Issue creation
* Tag creation

---

# 1️⃣ What is a Workflow Trigger?

A **trigger** is defined using the `on:` keyword inside the workflow YAML file.

## Basic Syntax

```yaml
name: My Workflow

on:
  push:
    branches:
      - main
```

This workflow runs whenever code is pushed to the `main` branch.

---

# 2️⃣ Common Workflow Events

| Event             | Purpose                           |
| ----------------- | --------------------------------- |
| push              | Runs when code is pushed          |
| pull_request      | Runs during PR creation/update    |
| workflow_dispatch | Manual trigger                    |
| schedule          | Run automatically on time         |
| release           | Trigger on GitHub releases        |
| issues            | Trigger when issue created/closed |
| fork              | Trigger when repo forked          |
| create            | Trigger when branch/tag created   |
| delete            | Trigger when branch/tag deleted   |
| watch             | Trigger when someone stars repo   |

---

# 3️⃣ Push Event

The most commonly used event.

## Example

```yaml
name: Push Workflow

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Print Message
        run: echo "Code pushed to main branch"
```

---

## Real-Time Use Case

Whenever developers push code:

* Run build
* Run unit tests
* Run security scan
* Deploy application

# 5️⃣ Manual Trigger — workflow_dispatch

Allows users to manually start workflow from GitHub UI.

## Example

```yaml
name: Manual Workflow

on:
  workflow_dispatch:

jobs:
  manual-job:
    runs-on: ubuntu-latest

    steps:
      - run: echo "Workflow triggered manually"
```

---

## How to Run

Go to:

* Repository
* Actions
* Select Workflow
* Click "Run workflow"

# 6️⃣ Scheduled Workflow (Cron Jobs)

Runs automatically based on time.

## Example

```yaml
name: Nightly Backup

on:
  schedule:
    - cron: '0 0 * * *'

jobs:
  backup:
    runs-on: ubuntu-latest

    steps:
      - run: echo "Running midnight backup"
```

---

# Cron Format

```text
* * * * *
| | | | |
| | | | Day of Week
| | | Month
| | Day
| Hour
Minute
```

---

## Example Cron Expressions

| Cron          | Meaning           |
| ------------- | ----------------- |
| `0 0 * * *`   | Daily midnight    |
| `*/5 * * * *` | Every 5 mins      |
| `0 9 * * 1`   | Every Monday 9 AM |

# 7️⃣ Multiple Triggers

One workflow can have multiple events.

## Example

```yaml
name: Multi Trigger Workflow

on:
  push:
    branches:
      - main

  pull_request:
    branches:
      - main

  workflow_dispatch:

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - run: echo "Workflow triggered"
```

---

# 8️⃣ Branch Filtering

Trigger only for specific branches.

## Example

```yaml
on:
  push:
    branches:
      - main
      - dev
```

Runs only for:

* main
* dev

---

# 9️⃣ Ignore Branches

## Example

```yaml
on:
  push:
    branches-ignore:
      - test
```

Workflow will NOT run for `test` branch.

---

# 🔟 Path Filtering

Run workflow only when specific files change.

## Example

```yaml
on:
  push:
    paths:
      - 'src/**'
```

Runs only when files inside `src` folder change.

---

# 1️⃣1️⃣ Ignore Paths

## Example

```yaml
on:
  push:
    paths-ignore:
      - 'README.md'
```

Workflow ignores README changes.
