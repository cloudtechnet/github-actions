# 28. Fail Fast Strategy

Fail Fast Strategy in GitHub Actions is used to:

> Stop unnecessary workflow execution immediately when failures occur.

This helps save:

* CI/CD execution time
* GitHub Actions minutes
* Runner resources
* Build costs

It is especially useful in:

* Matrix builds
* Large test pipelines
* Enterprise CI/CD systems

---

# What is Fail Fast?

Fail Fast means:

> As soon as one critical job fails, GitHub cancels remaining related jobs automatically.

Instead of waiting for all jobs to complete, the workflow stops early.

---

# Why Fail Fast is Important

Without Fail Fast:

```text id="n5e9lz"
10 test jobs running
1 job fails immediately
Remaining 9 continue running unnecessarily
```

This wastes:

* Time
* Resources
* CI minutes

---

# With Fail Fast

```text id="5k9o2g"
1 job fails
Remaining jobs cancelled automatically
```

Much faster and efficient.

---

# Where Fail Fast is Commonly Used

Mostly used with:

```yaml id="p2uqmz"
strategy:
  matrix:
```

because matrix builds create multiple parallel jobs.

---

# Default Behavior

In GitHub Actions:

> `fail-fast` is enabled by default for matrix jobs.

Meaning:

* One matrix job fails
* Other matrix jobs are cancelled automatically

---

# Syntax

```yaml id="6m3hvr"
strategy:
  fail-fast: true
```

---

# Example – Basic Fail Fast

```yaml id="kt7x7w"
name: Fail Fast Example

on: push

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      fail-fast: true

      matrix:
        node-version: [16, 18, 20]

    steps:
      - name: Display Node Version
        run: echo "Testing Node ${{ matrix.node-version }}"

      - name: Simulate Failure
        run: |
          if [ "${{ matrix.node-version }}" = "18" ]; then
            exit 1
          fi
```

---

# What Happens?

Matrix creates 3 jobs:

```text id="y7wrx8"
Node 16
Node 18
Node 20
```

When:

```text id="r0t7yk"
Node 18 fails
```

GitHub automatically cancels:

```text id="nnqjlwm"
Node 16 and Node 20 (if still running)
```

---

# Execution Flow

```text id="sqfntg"
Node 16 ─────►
Node 18 ──X FAILED
Node 20 ──CANCELLED
```

---

# Important Note

Fail-fast only affects:

✅ Matrix jobs

Not normal independent jobs.

---

# Matrix Strategy Overview

Matrix builds run multiple combinations automatically.

---

# Example

```yaml id="elz7se"
matrix:
  os: [ubuntu-latest, windows-latest]
  node: [16, 18]
```

Creates:

| OS      | Node |
| ------- | ---- |
| Ubuntu  | 16   |
| Ubuntu  | 18   |
| Windows | 16   |
| Windows | 18   |

4 parallel jobs.

---

# Fail Fast in Matrix Builds

Suppose:

```text id="8ecwkp"
Windows + Node 18 fails
```

Then:

* Remaining matrix jobs are cancelled

if:

```yaml id="jlwm0r"
fail-fast: true
```

---

# 1. Disabling Fail Fast

Sometimes you want ALL jobs to run even if one fails.

Use:

```yaml id="jlwm8d"
fail-fast: false
```

---

# Example

```yaml id="ut5m44"
name: Disable Fail Fast

on: push

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      fail-fast: false

      matrix:
        python-version: [3.8, 3.9, 3.10]

    steps:
      - name: Show Python Version
        run: echo "${{ matrix.python-version }}"

      - name: Simulate Failure
        run: |
          if [ "${{ matrix.python-version }}" = "3.9" ]; then
            exit 1
          fi
```

---

# Result

| Version | Status       |
| ------- | ------------ |
| 3.8     | ✅ Runs       |
| 3.9     | ❌ Fails      |
| 3.10    | ✅ Still Runs |

---

# Execution Flow

```text id="dzhum3"
3.8 ─────► SUCCESS
3.9 ──X FAILED
3.10 ─────► STILL RUNS
```

---

# Difference Between true and false

| fail-fast | Behavior                     |
| --------- | ---------------------------- |
| true      | Cancel remaining matrix jobs |
| false     | Continue all jobs            |

---

# 2. Real-World Use Cases

---

# Use Case 1 – Unit Testing

Enable fail-fast:

```yaml id="l9brk7"
fail-fast: true
```

Reason:

* One environment failure may indicate major issue
* Save CI time

---

# Use Case 2 – Cross-Platform Testing

Disable fail-fast:

```yaml id="vjlwm9"
fail-fast: false
```

Reason:

Need full report for:

* Linux
* Windows
* macOS

Even if one fails.

---

# Use Case 3 – Browser Testing

Disable fail-fast to see all browser issues.

Example:

* Chrome
* Firefox
* Safari

---

# 3. Fail Fast with Multiple Matrix Dimensions

---

# Example

```yaml id="ozqs5m"
strategy:
  fail-fast: true

  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [16, 18]
```

Creates 4 jobs.

If one fails:

* Remaining active jobs cancelled

# Matrix Execution Example

```text id="zwwjlwm"

Windows + 16 ──CANCELLED
Windows + 18 ──CANCELLED
```

---

# 4. Continue-on-error vs Fail Fast

These are different concepts.

---

# continue-on-error

Allows a job/step to fail without failing workflow.

---

# Example

```yaml id="ulmluc"
continue-on-error: true
```

---

# Difference

| Feature           | Purpose                      |
| ----------------- | ---------------------------- |
| fail-fast         | Cancel remaining matrix jobs |
| continue-on-error | Ignore failure               |

---

# Example

```yaml id="w0e1q7"
jobs:
  test:
    continue-on-error: true
```

Workflow continues even if job fails.

---

# 5. Combining Fail Fast and Continue-on-error

---

# Example

```yaml id="t4i4c7"
strategy:
  fail-fast: false
```

```yaml id="9sn3k0"
continue-on-error: true
```

Meaning:

* All matrix jobs run
* Failures don't stop workflow

Useful for:

* Experimental environments
* Beta testing

---

# 6. Real Enterprise Example

---

# Scenario

Company tests application on:

* Ubuntu
* Windows
* macOS

with:

* Node 16
* Node 18

---

# Workflow

```yaml id="r7lh7c"
name: Enterprise Matrix Testing

on: push

jobs:
  compatibility-test:
    runs-on: ${{ matrix.os }}

    strategy:
      fail-fast: false

      matrix:
        os:
          - ubuntu-latest
          - windows-latest
          - macos-latest

        node:
          - 16
          - 18

    steps:
      - name: Display Environment
        run: |
          echo "OS: ${{ matrix.os }}"
          echo "Node: ${{ matrix.node }}"

      - name: Run Tests
        run: echo "Testing application"
```

---

# Why fail-fast is false Here?

Because developers want complete compatibility report across all platforms.

---

# 7. Fail Fast in CI/CD Pipelines

---

# Typical Pipeline

```text id="epk79t"
Build
  ↓
Unit Tests
  ↓
Integration Tests
  ↓
Deployment
```

If tests fail early:

* Stop deployment immediately

This is fail-fast philosophy.

---

# Benefits of Fail Fast

| Benefit               | Description                 |
| --------------------- | --------------------------- |
| Faster feedback       | Detect issues quickly       |
| Save CI minutes       | Avoid unnecessary jobs      |
| Resource optimization | Fewer active runners        |
| Better debugging      | Failures visible early      |
| Faster pipelines      | Stop invalid builds quickly |

---

# 8. When NOT to Use Fail Fast

Disable fail-fast when:

* Need complete test report
* Testing multiple environments
* Gathering all compatibility issues
* Running experimental builds

---

# Example Situations

| Scenario                     | fail-fast |
| ---------------------------- | --------- |
| Production deployment        | true      |
| Cross-browser testing        | false     |
| Multi-platform compatibility | false     |
| Quick unit tests             | true      |

---

# 9. Advanced Example

```yaml id="95gxwv"
name: Advanced Fail Fast

on: push

jobs:
  matrix-test:
    runs-on: ${{ matrix.os }}

    strategy:
      fail-fast: true

      matrix:
        os:
          - ubuntu-latest
          - windows-latest

        python-version:
          - 3.9
          - 3.10

    steps:
      - name: Setup Python
        run: echo "Python ${{ matrix.python-version }}"

      - name: Simulate Failure
        run: |
          if [ "${{ matrix.python-version }}" = "3.10" ]; then
            exit 1
          fi
```

---

# Possible Result

```text id="nlc3gm"
Ubuntu + 3.9   → Running
Ubuntu + 3.10  → Failed
Windows + 3.9  → Cancelled
Windows + 3.10 → Cancelled
```

# 10. Best Practices

| Best Practice                               | Reason              |
| ------------------------------------------- | ------------------- |
| Use fail-fast for critical pipelines        | Save CI resources   |
| Disable fail-fast for compatibility testing | Get full reports    |
| Combine with matrix builds                  | Best efficiency     |
| Avoid unnecessary long-running jobs         | Faster feedback     |
| Use continue-on-error carefully             | Avoid hiding issues |

---

# 11. Common Mistakes

---

# Mistake 1 – Assuming Fail Fast Works for Normal Jobs

Fail-fast mainly affects matrix jobs.

---

# Mistake 2 – Using fail-fast for Debugging

During debugging:

```yaml id="kh9dxl"
fail-fast: false
```

is often better.

---

# Mistake 3 – Confusing with continue-on-error

These are completely different features.

---

# 12. Fail Fast vs Continue-on-error

| Feature          | Fail Fast      | Continue-on-error |
| ---------------- | -------------- | ----------------- |
| Cancels jobs     | Yes            | No                |
| Ignores failures | No             | Yes               |
| Matrix-specific  | Mostly yes     | No                |
| Purpose          | Save resources | Allow failures    |

---

# Summary

| Feature                 | Purpose                    |
| ----------------------- | -------------------------- |
| `fail-fast: true`       | Stop remaining matrix jobs |
| `fail-fast: false`      | Continue all matrix jobs   |
| Matrix strategy         | Parallel testing           |
| `continue-on-error`     | Ignore failures            |
| Early failure detection | Faster pipelines           |
