# 18. Workflow Dispatch Event

`workflow_dispatch` is a **manual trigger event** in GitHub Actions.
It allows you to start a workflow manually from:

* GitHub UI (Actions tab)
* GitHub CLI
* API
* External tools/scripts

This is useful when you want to run workflows **on demand** instead of automatically on every push or pull request.

---

# Why Use `workflow_dispatch`?

Normally workflows run automatically like:

* `push`
* `pull_request`
* `schedule`

But sometimes you need manual control.

Examples:

* Deploy application manually
* Run backup process
* Trigger production release
* Execute testing only when needed
* Select environment dynamically
* Run workflow with custom inputs

---

# Basic Syntax

```yaml
on:
  workflow_dispatch:
```

Example:

```yaml
name: Manual Workflow

on:
  workflow_dispatch:

jobs:
  demo:
    runs-on: ubuntu-latest

    steps:
      - name: Print Message
        run: echo "Workflow triggered manually!"
```

---

# How It Works

After pushing this workflow file:

1. Open GitHub repository
2. Go to **Actions**
3. Select workflow
4. Click **Run workflow**
5. Choose branch
6. Click **Run workflow** button

Now GitHub starts the workflow manually.

---

# Real-Time Example

## Example: Manual Deployment

```yaml
name: Deploy Application

on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Deploy App
        run: echo "Deploying application..."
```

Use case:

* Developer tests code
* Team lead manually deploys production
* Avoids automatic production deployments

---

# Workflow Dispatch With Inputs

You can ask users for values before running workflow.

Example:

```yaml
name: Environment Deployment

on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Select Environment'
        required: true
        default: 'dev'

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Show Environment
        run: echo "Deploying to ${{ github.event.inputs.environment }}"
```

---

# What Happens Here?

Before running workflow GitHub shows:

| Input       | Value             |
| ----------- | ----------------- |
| environment | dev / test / prod |

User selects value manually.

---

# Example Output

If user selects:

```text
prod
```

Output:

```text
Deploying to prod
```

---

# Multiple Inputs Example

```yaml
name: Manual Build

on:
  workflow_dispatch:
    inputs:
      app_name:
        description: 'Application Name'
        required: true

      version:
        description: 'Release Version'
        required: true

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Print Values
        run: |
          echo "App Name: ${{ github.event.inputs.app_name }}"
          echo "Version: ${{ github.event.inputs.version }}"
```

---

# Input Types

GitHub supports different input types.

Example:

```yaml
on:
  workflow_dispatch:
    inputs:
      deploy:
        type: boolean
        description: Deploy application?
        required: true

      environment:
        type: choice
        description: Choose environment
        options:
          - dev
          - test
          - prod
```

---

# Trigger Using GitHub CLI

Using GitHub CLI:

```bash
gh workflow run deploy.yml
```

With inputs:

```bash
gh workflow run deploy.yml -f environment=prod
```

---

# Trigger Using API

GitHub API can also start workflows.

Example:

```bash
curl -X POST \
-H "Accept: application/vnd.github+json" \
-H "Authorization: Bearer TOKEN" \
https://api.github.com/repos/OWNER/REPO/actions/workflows/main.yml/dispatches \
-d '{"ref":"main"}'
```

---

# Important Points

## 1. Workflow File Must Be in Default Branch

Usually:

* `main`
* `master`

Otherwise Run Workflow button may not appear.

---

## 2. User Needs Proper Permissions

Only users with repository access can manually run workflows.

---

## 3. Inputs Are Optional or Required

```yaml
required: true
```

or

```yaml
required: false
```

---

# Difference Between Push and Workflow Dispatch

| Feature        | push         | workflow_dispatch        |
| -------------- | ------------ | ------------------------ |
| Automatic      | Yes          | No                       |
| Manual Trigger | No           | Yes                      |
| User Inputs    | No           | Yes                      |
| Best For       | CI pipelines | Deployments/manual tasks |

# Complete Professional Example

```yaml
name: Production Deployment

on:
  workflow_dispatch:
    inputs:
      environment:
        description: "Deployment Environment"
        required: true
        type: choice
        options:
          - dev
          - staging
          - production

      version:
        description: "Application Version"
        required: true

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Source Code
        uses: actions/checkout@v4

      - name: Display Inputs
        run: |
          echo "Environment: ${{ github.event.inputs.environment }}"
          echo "Version: ${{ github.event.inputs.version }}"

      - name: Deploy Application
        run: echo "Deployment Started..."
```

# Summary

`workflow_dispatch` is used to:

✅ Run workflows manually
✅ Pass custom inputs
✅ Control deployments safely
✅ Execute workflows on demand
✅ Trigger workflows from UI, CLI, or API

It is one of the most commonly used events in GitHub Actions for production and DevOps workflows.
