# 21. Filtering Branches and Paths

In GitHub Actions, **Filtering Branches and Paths** helps you control **when a workflow should run**.

Instead of running workflows for every push or pull request, you can tell GitHub Actions:

* Run only for specific branches
* Run only when specific files/folders change
* Ignore some branches or files

This improves:

* Performance
* Faster CI/CD
* Reduced unnecessary workflow runs
* Better project organization

---

# 1. Branch Filtering

Branch filtering means:

> Trigger workflows only for selected branches.

---

# Why Use Branch Filters?

Example:

You may have branches like:

* `main`
* `develop`
* `feature/login`
* `feature/payment`

You may want:

* Deploy only from `main`
* Run testing on `develop`
* Ignore temporary branches

---

# Syntax

```yaml
on:
  push:
    branches:
      - main
```

This workflow runs only when code is pushed to the `main` branch.

---

# Example 1 – Run Workflow Only on Main Branch

```yaml
name: Main Branch Workflow

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Display Message
        run: echo "Code pushed to main branch"
```

---

# How It Works

If you push to:

| Branch        | Workflow Runs? |
| ------------- | -------------- |
| main          | ✅ Yes          |
| develop       | ❌ No           |
| feature/login | ❌ No           |

---

# Example 2 – Multiple Branches

```yaml
on:
  push:
    branches:
      - main
      - develop
```

Runs workflow on:

* `main`
* `develop`

---

# Example 3 – Wildcard Branch Matching

```yaml
on:
  push:
    branches:
      - 'feature/*'
```

Matches:

* `feature/login`
* `feature/payment`

Does NOT match:

* `bugfix/error`

---

# Wildcard Patterns

| Pattern     | Meaning                           |
| ----------- | --------------------------------- |
| `feature/*` | Any branch starting with feature/ |
| `release/*` | Any release branch                |
| `hotfix/*`  | Any hotfix branch                 |
| `**`        | Matches all branches              |

---

# Example 4 – Ignore Branches

Use `branches-ignore`.

```yaml
on:
  push:
    branches-ignore:
      - develop
```

Runs on all branches EXCEPT:

* `develop`

---

# Example 5 – Branch Filter with Pull Requests

```yaml
on:
  pull_request:
    branches:
      - main
```

Workflow runs when:

* Pull request targets `main`

Example:

```text
feature/login  --->  main
```

Workflow runs because target branch is `main`.

---

# Real-Time Use Case

## Scenario

You have:

* `main` → Production
* `develop` → Testing
* `feature/*` → Developer branches

You can configure:

| Branch    | Action      |
| --------- | ----------- |
| main      | Deploy      |
| develop   | Run tests   |
| feature/* | Run linting |

# 2. Path Filtering

Path filtering means:

> Run workflows only when specific files or folders change.

---

# Why Use Path Filters?

Suppose your repository contains:

```text
project/
 ├── frontend/
 ├── backend/
 ├── docs/
```

You may want:

* Frontend workflow only if frontend files change
* Backend workflow only if backend files change
* Ignore documentation changes

---

# Syntax

```yaml
on:
  push:
    paths:
      - 'frontend/**'
```

Workflow runs only if files inside `frontend` change.

---

# Example 1 – Frontend Changes Only

```yaml
name: Frontend Workflow

on:
  push:
    paths:
      - 'frontend/**'

jobs:
  frontend-build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - run: echo "Frontend files changed"
```

---

# How It Works

| Changed File       | Workflow Runs? |
| ------------------ | -------------- |
| frontend/app.js    | ✅ Yes          |
| frontend/style.css | ✅ Yes          |
| backend/server.js  | ❌ No           |

---

# Example 2 – Multiple Paths

```yaml
on:
  push:
    paths:
      - 'frontend/**'
      - 'backend/**'
```

Runs workflow if:

* frontend files change
  OR
* backend files change

---

# Example 3 – Ignore Paths

Use `paths-ignore`.

```yaml
on:
  push:
    paths-ignore:
      - 'docs/**'
```

Workflow runs for all changes EXCEPT documentation.

---

# Example 4 – File-Specific Filtering

```yaml
on:
  push:
    paths:
      - 'package.json'
```

Workflow runs only if:

```text
package.json
```

changes.

---

# Example 5 – Extension Filtering

```yaml
on:
  push:
    paths:
      - '**.js'
```

Runs when any JavaScript file changes.

Examples:

* `app.js`
* `frontend/main.js`
* `backend/server.js`

---

# 3. Combining Branch and Path Filters

You can combine both together.

---

# Example – Main Branch + Frontend Changes

```yaml
name: Frontend Production Workflow

on:
  push:
    branches:
      - main

    paths:
      - 'frontend/**'

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - run: echo "Deploying frontend from main branch"
```

---

# Workflow Runs Only If:

✅ Branch = `main`
AND
✅ Files changed inside `frontend/`

---

# Example Scenarios

| Branch  | File Changed    | Workflow Runs? |
| ------- | --------------- | -------------- |
| main    | frontend/app.js | ✅ Yes          |
| main    | backend/app.js  | ❌ No           |
| develop | frontend/app.js | ❌ No           |

---

# 4. Pull Request Filtering with Paths

```yaml
on:
  pull_request:
    branches:
      - main

    paths:
      - 'backend/**'
```

Workflow runs only when:

* PR target branch = `main`
  AND
* backend files changed

---

# 5. Advanced Wildcards

| Pattern | Meaning                          |
| ------- | -------------------------------- |
| `*`     | Matches any character except `/` |
| `**`    | Matches multiple directories     |
| `?.js`  | Single character match           |
| `[abc]` | Match a, b, or c                 |

---

# Examples

| Pattern     | Matches                |
| ----------- | ---------------------- |
| `src/*.js`  | src/app.js             |
| `src/**`    | All files under src    |
| `**/*.yml`  | All YAML files         |
| `docs/*.md` | Markdown files in docs |

# 7. Important Notes

---

## 1. Branch and Path Filters Work Together

Both conditions must match.

Example:

```yaml
branches:
  - main

paths:
  - 'src/**'
```

Workflow runs ONLY if:

* push happens on `main`
  AND
* files in `src` change

---

## 2. Filters Reduce Workflow Costs

Benefits:

* Less execution time
* Faster pipelines
* Reduced GitHub Actions usage

---

## 3. Useful for Monorepos

Large repositories often contain:

* frontend
* backend
* microservices
* infrastructure

Path filters help trigger only required workflows.

---

# 8. Best Practices

| Best Practice               | Why                           |
| --------------------------- | ----------------------------- |
| Use branch filters          | Avoid unnecessary deployments |
| Use path filters            | Run only required jobs        |
| Separate workflows          | Better maintainability        |
| Use wildcards carefully     | Prevent unexpected triggers   |
| Ignore docs/tests if needed | Save CI time                  |

---

# 9. Complete Example

```yaml
name: Smart CI Pipeline

on:
  push:
    branches:
      - main
      - develop

    paths:
      - 'src/**'
      - '.github/workflows/**'

    paths-ignore:
      - 'docs/**'

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Build Application
        run: echo "Building application..."
```

---

# What This Workflow Does

## Runs Only If:

### Branch is:

* `main`
  OR
* `develop`

AND

### Changed files are:

* inside `src/`
  OR
* workflow files

BUT

### Ignore:

* `docs/`
